## Deferred from: code review of 1-4+1-5 (2026-06-08)

- **Stranded terminal records are invisible to the roll-up** ([src/features/dashboard/useRollup.ts:68-101](../../src/features/dashboard/useRollup.ts#L68-L101)). Offboard (`OffboardDialog.tsx`) reassigns only *active* records and soft-deletes the subsidiary, leaving **terminal** records (churned customers, closed/resolved tickets, disqualified/converted leads) in the offboarded sub's bucket with `subsidiaryId === <deletedSubId>`. `useRollup` counts only **active** subsidiaries (`includeDeleted:false`) plus the parent (`subsidiaryId===null`) bucket, so those stranded records are counted by nothing → the tenant total **under-counts**, and a tenant whose only records are stranded shows a **false "empty"** state (amplified by the AC4 `grand===0` predicate). Pre-existing offboard×roll-up data-model interaction. **Action (product + E1/E3/E4):** decide whether offboarded subs' records should be counted in the roll-up, and/or whether offboard should reassign terminal records too (currently only active ones move). Severity: Medium.
- **`ROLE_LABELS[role]` → "undefined" for an unmapped role** ([src/features/dashboard/RollupPage.tsx:128](../../src/features/dashboard/RollupPage.tsx#L128)). The eyebrow's `ROLE_LABELS[session?.roles[0] ?? "viewer"]` has no not-found fallback; type-safe with the closed `Role` union today, but a future/forward-compat role string renders "undefined · …". **Action:** add a `?? <role-id>` fallback if `Role` ever widens. Severity: Low.
- **`scopeName` "Your subsidiary" fallback flashes for non-admins** ([src/features/dashboard/RollupPage.tsx:131-132](../../src/features/dashboard/RollupPage.tsx#L131-L132)). `currentSubName` resolves from `rows`, which are empty during load, so a non-admin briefly sees "… · Your subsidiary" before the real name (and it sticks if their sub never resolves into `rows`, e.g. the stranded/snap-back race). - **`useRollup.compute()` effect has no abort guard** ([src/features/dashboard/useRollup.ts:134](../../src/features/dashboard/useRollup.ts#L134)). The async `compute` can `setState` after unmount / on a superseded scope change (no `cancelled` flag/`AbortController`); the `set-state-in-effect` lint is suppressed to match the repo's load-effect convention. **Action:** add a cancellation guard when scope-switch races become observable. Severity: Low.

Severity: Low.

## Deferred from: code review of 0-12-notifications-kernel (2026-06-08)

- **No bus subscription in hook — incoming events don't trigger re-renders** (`useNotifications.ts`): `useNotifications()` reads `notificationsFor(recipient)` on render but has no subscription to the bus. A new `Ticket.Assigned` event arriving after mount won't update the displayed unread count until a user interaction forces a re-render. E5-S3 should add a `useEffect` subscribing to `publish` events and calling `forceUpdate()`. Severity: Medium.
- **`markRead` service function doesn't validate recipient ownership** (`NotificationService.ts:markRead`): `markRead(id)` finds by id only; no check that the notification's `recipient` matches the caller. Low risk in the current single-user session model but should be tightened before multi-tab or shared-session scenarios. Severity: Low.
- **No dedup guard for duplicate `eventId` in `_store`** (`NotificationService.ts:handleEvent`): `publish()` can be called multiple times with the same `eventId`; `_store.push` has no uniqueness check. Duplicate notifications with the same `id` will cause `markRead` to leave the second copy unread. Fix when idempotency guarantees are needed (E6/outbox). Severity: Low.
- **`markRead`/`markAllRead` recreated on every render** (`useNotifications.ts`): plain function declarations inside render body create new references every render; wrap in `useCallback` in E5-S3 to prevent downstream `React.memo` thrash. Severity: Low.
- **HMR double subscription in dev** (`NotificationService.ts`): Vite HMR re-evaluates the module with a fresh `_unsubscribe = null`, so the old subscription from the prior module instance is never removed. Two `handleEvent` closures run simultaneously in dev after a hot reload. Severity: Low (dev-only).

## Deferred from: code review of 0-2-author-baseentity-and-canonical-entity-types (2026-06-07)

- **`crypto.randomUUID()` requires a secure context** ([src/shared/domain/types.ts:62](../../src/shared/domain/types.ts#L62)). `newId` calls the bare global `crypto.randomUUID()`. It resolves in the Vitest node environment, on `localhost` dev, and over HTTPS in production — but in a browser it is exposed **only in a secure context**, so serving the pilot over plain `http://` on a LAN IP (e.g. demoing from another device) makes `crypto.randomUUID` `undefined` → `TypeError` on every entity create. The story explicitly accepted "no polyfill, no `uuid` dependency" (Dev Notes), so this is **not** a code change for E0-S2. **Action for whoever owns deployment / the dev-server story:** ensure the app is always served from a secure context (HTTPS or `localhost`), or add a guarded fallback at that point. Severity: Medium (only triggers in an insecure-context deployment).

## Deferred from: code review of 0-3-define-repository-page-listquery (2026-06-07)

- **No explicit `"strict": true` in any tsconfig** ([tsconfig.app.json](../../tsconfig.app.json)). The seam's nullability/optionality contracts (`get(): Promise<T | null>`, the `?:` vs `| null` distinction) rely on `strictNullChecks` being **on by default in TypeScript 6.x** — verified empirically active (`tsc` flags `null`→`string` with no flags, v6.0.3). This is **not** an E0-S3 issue (the contract is correct under the current toolchain) and is project-wide. **Action for whoever owns the build config:** add `"strict": true` to `tsconfig.app.json` so the load-bearing nullability contracts are pinned explicitly rather than depending on a compiler default that a TS-version or scaffold change could flip. Severity: Low (latent; only bites if the strict default changes).

## Deferred from: code review of 0-7-audit-log-domain-event-bus-shared-correlationid (2026-06-07)

- **localStorage write-through silently diverges from memory on a `bigint` payload** ([src/shared/events/auditLog.ts:310-318](../../src/shared/events/auditLog.ts#L310-L318)). `persist()` serializes the whole store with `JSON.stringify(store)`, which throws `TypeError` on a bigint value; the `catch` swallows it, so this append (and the entire prior store) silently fails to persist while the in-memory array stays correct. In-memory is the documented pilot source of truth, so this only affects the browser durable mirror. **Action for whoever owns real persistence (E0-S4 / backend adapter):** decide on a serialization policy for non-JSON values (reject at validation, coerce, or a custom replacer) when the audit stream gets a durable store. Severity: Low (pilot in-memory unaffected).
- **`bus.publish` hands the same mutable event object to every subscriber** ([src/shared/events/bus.ts:506-515](../../src/shared/events/bus.ts#L506-L515)) despite the `DomainEvent` doc comment calling it "immutable." A subscriber can mutate `payload`/`correlationId`, contaminating later subscribers and potentially the correlationId after the audit was appended. **Action:** freeze/clone the published event (as the audit log already does) if/when event handlers are no longer fully trusted. Severity: Low (single-trust synchronous pilot).

## Deferred from: code review of 0-6-route-guard-action-guard-permission-matrix (2026-06-08)

- **The authZ gates/predicates never validate session `exp`** ([src/shared/auth/permissions.ts](../../src/shared/auth/permissions.ts)). `can`/`resolveOutcome`/`authorize` and both guards read `roles`/scope from the claims but never compare `exp` to now, so a present-but-expired session authorizes normally. This is the same gap already logged for E0-S5 (`isAuthenticated` ignores `exp`); the mock `2099` constant never expires. **Action for the OIDC swap (Epic 6):** enforce `exp` once real tokens arrive, ideally at the kernel (`useAuth`/`isAuthenticated`) so every gate inherits it. Severity: Low (latent; masked by the mock constant).
- **`authorize()` propagates a subscriber `AggregateError` rather than returning its `AuthZOutcome`** ([src/shared/auth/permissions.ts](../../src/shared/auth/permissions.ts)). `emitRoleDenied` appends the audit then `publish()`es; if any future `Auth.RoleDenied` subscriber throws, `bus.publish` re-raises an `AggregateError` after delivery, so `authorize()` throws to its caller (E0-S4's authorize beat) instead of returning `"denied"` — the audit record exists, but the documented return contract breaks on that path. This mirrors the ratified append-before-publish reference pattern (AuthProvider) and the E0-S7 bus "handler faults surface, not swallowed" decision; no `Auth.RoleDenied` subscribers exist in the pilot. **Action (with the E0-S7 bus hardening):** isolate/clone handler faults (or wrap the publish) when event handlers are no longer fully trusted so a buggy listener can't convert a clean denial into a thrown error. Severity: Low (single-trust synchronous pilot; no subscribers yet).
