# E0-S6 ‚Äî Route guard + action guard from the permission matrix (ADR-015 predicates)

> **GitHub Epic ‚Üí Feature ‚Üí Story:** Epic **E0 ‚Äî Platform Guidelines & Standards** ‚Ä∫ Feature **0.3 ‚Äî Auth & RBAC Kernel** ‚Ä∫ Story **E0-S6**
> **Cut:** Pilot ¬∑ **Depends on:** E0-S5 (claims), E0-S7 (audit bus for `Auth.RoleDenied`)

## Story
As the platform, two authorization gates (screen access and per-mutation access) decide every grant from one data-encoded permission matrix with predicate semantics, so that no feature invents its own access logic and deny always wins.

## Implementation-Ready Acceptance Criteria
1. **AC1 ‚Äî Two gates.** A **route guard** decides screen access; an **action guard** decides per-mutation access on a specific record within the caller's tenant/subsidiary. Both delegate to one `can(actor, capability, action, resource)` predicate.
   *Touches:* `src/shared/auth/guards.tsx`, `src/shared/auth/permissions.ts` ¬∑ *Seam:* authZ guards ¬∑ *Schema/Map:* Grant table.
2. **AC2 ‚Äî Deny-wins, default deny.** Conflicting or absent grants resolve to deny; the default grant for any unlisted `(capability, role)` cell is `deny`.
   *Touches:* `src/shared/auth/permissions.ts` ¬∑ *Seam:* `can()` predicate ¬∑ *Schema/Map:* Grant table.
3. **AC3 ‚Äî ¬ß2.2 matrix encoded as data, cell-tested.** The ¬ß2.2 permission matrix (as **amended** ‚Äî see DEC-CC-1) is encoded as a `Grant` table (`Capability √ó Role ‚Üí Grant`) and unit-tested cell-by-cell. **Ticket `create` is `allow` for all four roles** (`tenant_admin`/`sales`/`support`/`viewer`) per **DEC-CC-1**, while ticket **edit/assign** stays `allow` for `tenant_admin`/`support` and `view` for `sales`/`viewer`. **Customer `create/edit`** is `allow` for `tenant_admin`/`sales`, `view` for `support`/`viewer` (C-2 kept).
   *Touches:* `src/shared/auth/permissions.ts`, `src/shared/auth/permissions.test.ts` ¬∑ *Seam:* `can()` predicate ¬∑ *Schema/Map:* Grant table (incl. amended ticket-create row, DEC-CC-1).
4. **AC4 ‚Äî Out-of-tenant ‚Üí 404 (UC-5).** Access to a record outside the caller's tenant returns `404` (existence is never disclosed), not `403`.
   *Touches:* `src/shared/auth/guards.tsx`, `src/shared/auth/permissions.ts` ¬∑ *Seam:* authZ guards ¬∑ *Schema/Map:* n/a.
5. **AC5 ‚Äî Denied action audited (UC-2).** A denied action emits `Auth.RoleDenied` to the dual streams with a `correlationId`.
   *Touches:* `src/shared/auth/permissions.ts` (+ `shared/events/auditLog.ts`, `shared/events/bus.ts`) ¬∑ *Seam:* `can()` + event/audit bus ¬∑ *Schema/Map:* `Auth.RoleDenied`.
6. **AC6 ‚Äî ADR-015 predicate model (DELTA).** Encode predicates: `Grant ‚àà allow|deny|view|own|restricted`. `own` ‚áí `isOwned(actor, resource)` = (`ownerId|assigneeId|actorId === userId`). `restricted` ‚áí `isOwned AND action ‚àà {softDelete, export}`. **Hard-delete is NEVER granted** (tenant_admin only). Concretely: "View audit/events" ‚Üí `own` (sales/support), `allow` (tenant_admin), `deny` (viewer); "Delete(soft)/export" ‚Üí `restricted` (sales/support).
   *Touches:* `src/shared/auth/permissions.ts`, `src/shared/auth/permissions.test.ts` ¬∑ *Seam:* `can()` predicate ¬∑ *Schema/Map:* Grant table + predicates.
7. **AC7 ‚Äî Authentication gate: public `/sign-in`, recognize session before redirecting (DEC-CC-7).** Screen access requires an **established session** in addition to the capability check. An **unauthenticated** visit to any guarded route **redirects to the public `/sign-in`** route (not a 404/blocked state ‚Äî being signed out is not an out-of-tenant denial), carrying the intended destination as `from` so sign-in can return the user there after login. `/sign-in` itself is **public/ungated** and reachable while signed out. The guard **recognizes an authenticated session** (the redirect fires only when `session === null`); once a session exists the capability gate (AC1) decides the screen. **Distinct from AC4's out-of-tenant `404`:** authentication failure ‚Üí redirect to `/sign-in`; *authorization* failure (wrong role / out-of-tenant) ‚Üí blocked/`404` per AC2/AC4.
   *Touches:* `src/app/router.tsx` (`RequireAuth`, `SignInPage`), `src/shared/auth/guards.tsx` ¬∑ *Seam:* route guard + authentication redirect ¬∑ *Schema/Map:* n/a.

## Test Requirements (NFR-12)
- **Vitest:** the cell-by-cell matrix test enumerates **every** `(capability, role)` cell against fixtures for owned/not-owned, read/mutate, and soft/hard-delete; asserts `own` requires ownership, `restricted` allows only `{softDelete, export}` on owned records, hard-delete is denied for all roles incl. sales/support, default cells deny, and the concrete ¬ß2.2 rows resolve as specified ‚Äî including **ticket `create` = allow for all four roles (DEC-CC-1)** with ticket **edit/assign** still denied to `sales`/`viewer`; **customer `create/edit` = allow only for `tenant_admin`/`sales`, denied to `support`/`viewer` (C-2)**; "View audit/events" (`own` sales/support, `allow` admin, `deny` viewer); "Delete(soft)/export" (`restricted` sales/support).
- **RTL:** route guard renders/blocks a screen by role; action guard disables/permits a mutation control on owned vs. not-owned records.
- **Playwright:** denial + out-of-tenant 404 paths exercised by the cross-tenant isolation E2E (E0-S11).
