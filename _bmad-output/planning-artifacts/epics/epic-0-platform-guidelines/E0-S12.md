# E0-S12 — Notifications kernel

> **GitHub Epic → Feature → Story:** Epic **E0 — Platform Guidelines & Standards** › Feature **0.7 — Notifications Kernel** › Story **E0-S12**
> **Cut:** Pilot · **Depends on:** E0-S7 (domain-event bus), E0-S9 (four-state surface)

## Story
As the platform, in-app notifications are a shared-layer capability driven off the domain-event bus — not feature-owned — so that any event can produce a scoped, recipient-targeted notification without bespoke writes.

## Implementation-Ready Acceptance Criteria
1. **AC1 — Service + hook.** `src/shared/notifications` ships a `NotificationService` that **subscribes to the `DomainEvent` bus** (E0-S7) and a `useNotifications()` hook for consumers.
   *Touches:* `src/shared/notifications/NotificationService.ts`, `src/shared/notifications/useNotifications.ts` · *Seam:* notifications kernel subscribing to event bus · *Schema/Map:* `DomainEvent` (E0-S7).
2. **AC2 — Notification data shape.** A notification carries `{ recipient, type, payload, read/unread, tenantId, subsidiaryId }` — scoped to the recipient's tenant/subsidiary.
   *Touches:* `src/shared/notifications/NotificationService.ts` · *Seam:* notifications kernel · *Schema/Map:* notification shape.
3. **AC3 — Recipient-mapping handlers (UC-5).** Handlers map event type → recipient (e.g. `Ticket.Assigned → assigneeId`; `Lead.Converted → lead ownerId`), scoped to the recipient's tenant/subsidiary. Notifications are driven off domain events **only — never bespoke writes**.
   *Touches:* `src/shared/notifications/NotificationService.ts` · *Seam:* notifications kernel + event bus · *Schema/Map:* event-type → recipient map.
4. **AC4 — Four-state surface scaffold (UC-1).** A four-state notifications surface scaffold (loading/empty/error/data via `QueryStateBoundary`) is provided, consumed later by E5-S3 / E4-S3.
   *Touches:* `src/shared/notifications/useNotifications.ts` (+ `src/shared/ui/QueryStateBoundary.tsx`) · *Seam:* four-state boundary · *Schema/Map:* n/a.

## UX & Behavior (from prototype)
*Source: `prototype/app/shell.jsx` (`NotificationsBell`), `prototype/app/dashboard.jsx` (`DashboardNotifications`, `NotificationsPage`). The kernel + scaffold must support the three realized surfaces below (the surfaces themselves ship in E4-S3/E5-S3). Visual oracle: `dashboard.png`, `cust-detail.png` (bell badge "2").*

- **One model, three surfaces** — all read `notificationsFor(user)`, **newest-first**, unread rows tinted `--iso-blue-3-50` with an unread **dot**: (1) topbar **bell** dropdown, (2) **dashboard** notifications card, (3) full **Notifications page**.
- **Notification shape:** `{ recipient, type, payload, read/unread, tenantId, subsidiaryId }`, scoped to the recipient's tenant/subsidiary (UC-5).
- **Bell:** unread **count badge** in `--iso-danger`; dropdown lists items + **"Mark all as read"** + per-item unread dot + footer **"View all notifications →"**. New notification animates the badge at `--crm-fast`.
- **Per-item actions:** "Mark as read" + "Mark all as read"; **relative time** ("X min ago"), absolute on hover. **Notifications persist until read** (a toast is ephemeral; these are not).
- **Recipient mapping (lives here, not in features):** `Ticket.Assigned → assigneeId`, `Lead.Converted → lead ownerId` — event-driven off the bus only, **never bespoke writes**.
- **Four states** (the scaffold): loading (skeleton) / **empty = "You're all caught up" (`bell-off`)** / error (retry) / ready — via `QueryStateBoundary`.
- **Seeded examples** (for demo realism): Lena Bauer (ticket assigned / high-priority), Marco Ruiz (lead added), Sara Khan (merge completed).
