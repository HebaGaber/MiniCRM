# E0-S7 — Audit log + domain event bus with shared correlationId

> **GitHub Epic → Feature → Story:** Epic **E0 — Platform Guidelines & Standards** › Feature **0.4 — Event, Audit & Logging Kernel** › Story **E0-S7**
> **Cut:** Pilot · **Depends on:** E0-S1 (entity vocabulary for the event registry)

## Story
As the platform, every operation emits one canonical domain event and writes one immutable audit record on a shared `correlationId`, so that activity timelines, audit logs, and notifications all derive from one trustworthy stream.

## Implementation-Ready Acceptance Criteria
1. **AC1 — Event & audit shapes.** Define `AuditEvent` and `DomainEvent` per constitution §7 (incl. `type`, `entityType`, `entityId`, `tenantId`, `subsidiaryId|null`, `actorId`, `correlationId`, `ts`; `AuditEvent` adds `before`/`after`).
   *Touches:* `src/shared/events/bus.ts`, `src/shared/events/auditLog.ts` · *Seam:* event bus + audit log · *Schema/Map:* `DomainEvent`/`AuditEvent`.
2. **AC2 — Append-only audit.** The audit log is append-only — records are never updated or deleted.
   *Touches:* `src/shared/events/auditLog.ts` · *Seam:* audit log · *Schema/Map:* `AuditEvent`.
3. **AC3 — Canonical event-type registry.** A registry enforces `PascalCaseEntity.PastTenseAction` names (Lead.Created/Updated/StatusChanged/Converted/Deleted; Customer.*; Ticket.Created/Updated/StatusChanged/Assigned/Deleted; Tenant.SubsidiaryAdded/Removed; Auth.*). Free-form names are rejected at publish time.
   *Touches:* `src/shared/events/eventTypes.ts`, `src/shared/events/bus.ts` · *Seam:* event bus registry · *Schema/Map:* event-type registry.
4. **AC4 — One op → one event + one audit, same correlationId (UC-2).** A single operation produces exactly one `DomainEvent` and one `AuditEvent` sharing one `correlationId`; ship an **assertable test helper** that verifies this.
   *Touches:* `src/shared/events/bus.ts`, `src/shared/events/auditLog.ts`, `src/shared/events/correlation.ts` · *Seam:* event/audit bus + correlation · *Schema/Map:* n/a.
5. **AC5 — Redaction in before/after.** Secrets and PII are redacted in audit `before`/`after` snapshots.
   *Touches:* `src/shared/events/auditLog.ts` · *Seam:* audit log · *Schema/Map:* redaction rules (shared with E0-S8 masking).

## Architecture — components, seams & modules touched
- **Modules (NFR-11):** `src/shared/events/bus.ts`, `src/shared/events/auditLog.ts`, `src/shared/events/eventTypes.ts`, `src/shared/events/correlation.ts`.
- **Patterns:** dual streams (in-process bus + append-only audit), one `correlationId`, canonical registry; **not** event-sourcing. Feeds the 4-beat *emit*/*audit* beats (E0-S4), ADR-016 activity-timeline vs audit-log split, and the notifications kernel (E0-S12 subscribes to the bus).
- **ADRs:** ADR-008 (dual streams, registry, correlationId).

## Test Requirements (NFR-12)
- **Vitest:** registry accepts canonical names and rejects free-form/malformed names; audit log rejects update/delete (append-only); the test helper asserts one op → one domain event + one audit event with an identical `correlationId`; redaction strips configured secret/PII keys from `before`/`after`.
- **RTL:** n/a.
- **Playwright:** n/a (consumed by timeline/audit panels and notifications in later epics).
