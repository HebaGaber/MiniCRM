# E0-S3 — Define Repository<T> + Page<T> + ListQuery

> **GitHub Epic → Feature → Story:** Epic **E0 — Platform Guidelines & Standards** › Feature **0.2 — Repository Seam & Data Kernel** › Story **E0-S3**
> **Cut:** Pilot · **Depends on:** E0-S2 (entity types)

## Story
As the platform, all data access goes through one generic `Repository<T>` seam with a uniform paginated list contract, so that features never touch persistence directly and the adapter can swap without changing call sites.

## Implementation-Ready Acceptance Criteria
1. **AC1 — `Repository<T>` interface.** Define the interface per constitution §4.1: `list(q?: ListQuery): Promise<Page<T>>`, `get(id): Promise<T | null>`, `create(input): Promise<T>`, `update(id, patch, version): Promise<T>`, `remove(id): Promise<void>` (soft). No persistence detail leaks into the type.
   *Touches:* `src/shared/data/Repository.ts` · *Seam:* `Repository<T>` · *Schema/Map:* `T extends BaseEntity` (E0-S2).
2. **AC2 — `Page<T>` paging contract.** `Page<T>` returns `{ items: T[], total, page, pageSize }` with default `pageSize=25` and **max 100** (requests above the max are clamped to 100, not errored).
   *Touches:* `src/shared/data/Repository.ts` · *Seam:* `Repository<T>` · *Schema/Map:* `Page<T>`.
3. **AC3 — `ListQuery` standard params.** `ListQuery` accepts the standard query params `q`, `status`, `ownerId`, `page`, `pageSize`, `sort`; **unknown filters are ignored, never errored** (UC-5 forward-compat).
   *Touches:* `src/shared/data/Repository.ts` · *Seam:* `Repository<T>` · *Schema/Map:* `ListQuery`.
