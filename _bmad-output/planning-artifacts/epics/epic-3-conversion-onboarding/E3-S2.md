# E3-S2 — Walk a customer prospect → onboarding → active

> **GitHub Epic → Feature → Story:** Epic **E3 — Conversion Saga & Customer Onboarding** › Feature **3.2 — Customer Onboarding Workflow** › Story **E3-S2**
> **Cut:** Pilot · **Depends on:** **E0-S1** (`CUSTOMER_TRANSITIONS`), **E0-S2** (`WorkflowInstance` + `WorkflowRunner`) — *hard prerequisites* · **E0-S4** (repository/4-beat), **E0-S6** (action guard), **E0-S7** (dual-event bus) · E3-S1 (a `prospect` customer exists to onboard)

## Story
As a salesperson/admin, I run the onboarding workflow that activates a new customer, so that a freshly converted prospect is walked through a tracked, legal-only state progression to `active`.

## Implementation-Ready Acceptance Criteria

1. **AC1 — Only `CUSTOMER_TRANSITIONS` moves allowed; illegal → `422` (UC-3).** Every state move is validated by `canTransition` against `CUSTOMER_TRANSITIONS`; a move not in the map is rejected `422`, the `StatusPill` is unchanged, and **no** `StatusChanged` event is emitted. State moves go through `POST /customers/{id}/transition` — never a `PATCH status`.
   *Touches:* `src/features/customers/onboarding/onboardingWorkflow.ts`, `src/shared/domain/status.ts` · *Seam:* service-layer `canTransition` precondition + transition endpoint · *Schema/Map:* `CUSTOMER_TRANSITIONS`.

2. **AC2 — Legal state model (DEC-1).** `prospect → onboarding → active`, with `active ↔ inactive` reactivation and `active`/`inactive → churned` (`churned` terminal). Exactly the `CUSTOMER_TRANSITIONS` map:
   `prospect→[onboarding]; onboarding→[active]; active→[inactive,churned]; inactive→[active,churned]; churned→[]`.
   *Touches:* `src/shared/domain/status.ts`, `src/features/customers/onboarding/onboardingWorkflow.ts` · *Seam:* transition map enforcement · *Schema/Map:* `CUSTOMER_TRANSITIONS` (E0-S1).

3. **AC3 — Built on the shared Tasks/Workflow seam (proves the seam a second time).** Onboarding is a `WorkflowInstance` of `type: "customer_onboarding"` driven by the same `WorkflowRunner` (E0-S2) used by the conversion saga (E3-S1) — reusing, not re-implementing, the engine.
   *Touches:* `src/features/customers/onboarding/onboardingWorkflow.ts`, `src/shared/workflow/WorkflowRunner.ts` · *Seam:* `WorkflowRunner` over `Repository<WorkflowInstance>` · *Schema/Map:* `WorkflowInstance` (`type: "customer_onboarding"`).

4. **AC4 — Each accepted change emits `Customer.Updated` (+ StatusChanged semantics) + audit, one `correlationId` (UC-2).** An accepted transition is a standard 4-beat mutation emitting one `Customer.Updated` `DomainEvent` (carrying StatusChanged semantics) + one immutable `AuditEvent`, sharing one `correlationId` minted at action start.
   *Touches:* `src/features/customers/onboarding/onboardingWorkflow.ts`, `src/shared/events/{bus.ts, auditLog.ts, correlation.ts}` · *Seam:* 4-beat mutation (authorize→mutate→emit→audit) · *Schema/Map:* canonical event `Customer.Updated`.

## UX & Behavior (from prototype)
*Source: `prototype/app/store.jsx` (`TRANSITIONS.customer`, `transitionCustomer`), `prototype/app/shared_records.jsx` (`ChangeStatusControl`). Build identical to the prototype.*

- **Transitions use the shared `ChangeStatusControl`** (see E2-S3): legal-only next states as `→` pills under "Move this customer to"; illegal states shown **locked** with an inline rule panel; **no pill change / no event** on a blocked move (422 discipline). Customer tones: prospect `neutral`, onboarding `info`, active `success`, inactive `warning`, churned `danger`. Map matches constitution §3.2 exactly: `prospect→[onboarding]`, `onboarding→[active]`, `active→[inactive,churned]`, `inactive→[active,churned]`, `churned→[]`.
- **✅ C-3 RESOLVED (DEC-CC-2) — Activation gate is BINDING.** `onboarding → active` is **blocked unless the customer has both `taxRegistrationNumber` and `contactAddress`** (§2.2 fields, added per DEC-CC-2). The block surfaces **inline in the change-status control** as *"Add a tax registration number and contact address before activating this customer. Edit the customer to fill it in, then activate."* — **no pill change, no `*.StatusChanged` event** until satisfied (a customer-state precondition analogous to the ticket gate). Implemented in `transitionCustomer` as a precondition on the `onboarding→active` edge; `CUSTOMER_TRANSITIONS` is structurally unchanged. **This is now an acceptance criterion** (it refines AC1/AC2 for the activation edge), not a deferral.
- **C-3 test addendum:** assert `onboarding → active` is rejected `422` (inline, no event, pill unchanged) when either field is missing, and accepted once both are present.
- **Motion:** accepted transition optimistic apply `--crm-instant`, pill tone cross-fade `--crm-fast`, rollback `--crm-slow`.

## Test Requirements (NFR-12)
- **Vitest:** `CUSTOMER_TRANSITIONS` table — every legal move accepted, every illegal move (e.g. `prospect→active`, `churned→*`, `active→prospect`) → `422` with pill unchanged and no `StatusChanged` event; reactivation `inactive→active`; `→churned` terminal; one `Customer.Updated` + one `AuditEvent` per accepted move sharing one `correlationId`; onboarding `WorkflowInstance` uses the same `WorkflowRunner` as conversion.
- **RTL:** onboarding step UI reflects legal next-states only; transition action is optimistic with rollback on error.
- **Playwright:** prospect → onboarding → active happy path; illegal-move attempt surfaces `422` without changing the pill.
