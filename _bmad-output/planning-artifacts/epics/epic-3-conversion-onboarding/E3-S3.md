# E3-S3 — Customer list/detail with lineage + timeline

> **GitHub Epic → Feature → Story:** Epic **E3 — Conversion Saga & Customer Onboarding** › Feature **3.3 — Customer List & Detail** › Story **E3-S3**
> **Cut:** Pilot · **Depends on:** **E0-S1** (`CUSTOMER_TRANSITIONS` / `STATUS_TONE`) · **E0-S4** (repository reads/4-beat), **E0-S6** (record-view permission gate), **E0-S7** (domain-event stream the timeline reads) · E3-S1 (customer record + lineage pointers)

## Story
As a user, I view customers and trace each back to its originating lead, so that I can see a customer's full provenance and activity in one place.

## Implementation-Ready Acceptance Criteria

1. **AC1 — `ListPage` + `DetailPage` with customer-toned `StatusPill`.** Customers render on the shared `ListPage` template and a `DetailPage`; the `StatusPill` shows the four customer states including `prospect`/`onboarding`, colored only from `STATUS_TONE` (no hex/px) (NFR-10).
   *Touches:* `src/features/customers/CustomerListPage.tsx`, `src/features/customers/CustomerDetailPage.tsx`, `src/shared/ui/{templates/ListPage, templates/DetailPage, components/StatusPill}` · *Seam:* `Repository<Customer>.list/get` · *Schema/Map:* `STATUS_TONE` for customer states.

2. **AC2 — Detail shows lineage + activity timeline (ADR-016).** The DetailPage shows **lineage** — a link to the originating lead via `convertedFromLeadId` — and the **Activity timeline** = the `DomainEvent` stream filtered by `entityType=Customer + entityId`, **gated by permission to view the record** (ADR-016 timeline split; the viewer is included within scope).
   *Touches:* `src/features/customers/CustomerDetailPage.tsx`, `src/features/tickets/ActivityTimeline.tsx` (shared component), `src/shared/auth/permissions.ts` · *Seam:* `DomainEvent` stream read + `can()` record-view gate · *Schema/Map:* event filter `entityType+entityId`.

3. **AC3 — Related Tickets tab (feeds Epic 4).** The DetailPage exposes a related Tickets tab scoped to the customer; in this story it is the seam/placeholder that Epic 4 populates (no ticket CRUD here).
   *Touches:* `src/features/customers/CustomerDetailPage.tsx` · *Seam:* `Repository<Ticket>.list({ customerId })` (consumed by Epic 4) · *Schema/Map:* n/a.

4. **AC4 — Four states, scoped (UC-1, UC-5).** List and detail render loading/empty/error/data via `<QueryStateBoundary>`; all reads are scoped from auth context only — out-of-tenant access → `404` (no existence leak).
   *Touches:* `src/features/customers/{CustomerListPage.tsx, CustomerDetailPage.tsx}`, `src/shared/ui/QueryStateBoundary.tsx`, `src/shared/auth/guards.tsx` · *Seam:* `Repository<Customer>` scoped reads + route guard · *Schema/Map:* n/a.

- **UC-5 — Tenant scope.** Customer reads scoped from auth context only; out-of-tenant → `404`.

- **Modules (NFR-11):** `src/features/customers/{CustomerListPage.tsx, CustomerDetailPage.tsx}`; `src/features/tickets/ActivityTimeline.tsx`; `src/shared/ui/{templates/ListPage, templates/DetailPage, components/StatusPill, QueryStateBoundary.tsx}`; `src/shared/auth/{permissions.ts, guards.tsx}`.

## Test Requirements (NFR-12)
- **Vitest:** customer-state → `STATUS_TONE` mapping (incl. `prospect`/`onboarding`); lineage resolver returns the correct originating lead from `convertedFromLeadId`; timeline filter selects only `Customer/{id}` DomainEvents; record-view gate (`can()`) admits in-scope viewer, denies out-of-scope.
- **RTL:** list + detail render four states; lineage link navigates to the originating lead; Tickets tab renders as an (empty) seam; StatusPill uses tones not hex.
- **Playwright:** open a converted customer, follow the lineage link back to its lead, confirm the linked conversion event appears on the customer timeline; out-of-tenant customer URL → `404`.
