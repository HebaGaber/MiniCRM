# E3-S1 ‚Äî Convert a qualified lead via the reference saga

> **GitHub Epic ‚Üí Feature ‚Üí Story:** Epic **E3 ‚Äî Conversion Saga & Customer Onboarding** ‚Ä∫ Feature **3.1 ‚Äî Lead ‚Üí Customer Conversion Saga** ‚Ä∫ Story **E3-S1**
> **Cut:** Pilot ¬∑ **Depends on:** **E0-S1** (`CUSTOMER_TRANSITIONS`), **E0-S2** (`WorkflowInstance` + `WorkflowRunner`) ‚Äî *hard prerequisites, must land first* ¬∑ **E0-S4** (repository/4-beat per step), **E0-S6** (action guard for the convert/guard step), **E0-S7** (dual-event bus for the conversion events) ¬∑ E2-S2 (lead can reach `qualified`)

## Story
As a salesperson, I convert a `qualified` lead into a `prospect` customer through a persisted, resumable, compensating saga, so that conversion is auditable, never leaves a half-made customer, and links the customer back to its originating lead.

## Implementation-Ready Acceptance Criteria

1. **AC1 ‚Äî Conversion legal only from `qualified` (UC-4).** The `guard` step asserts the lead is `qualified` **and** not already converted (`convertedToCustomerId == null`); a lead in `new`/`contacted`/`disqualified` (or already `converted`) cannot convert and re-conversion is blocked ‚Äî illegal entry ‚Üí `422` (no saga started, no mutation).
   *Touches:* `src/features/customers/conversion/conversionSaga.ts`, `src/shared/domain/status.ts` ¬∑ *Seam:* action-guard precondition (step 1, read-only) ¬∑ *Schema/Map:* `LEAD_TRANSITIONS` (qualified‚Üíconverted is the only legal source) + field-map check.

2. **AC2 ‚Äî Saga is a persisted `WorkflowInstance`, NOT an inline function (DEC-2).** The conversion is a `WorkflowInstance` of `type: "lead_conversion"` (`{ id, status: running|completed|compensating|failed, currentStep, steps[], completedSteps[], correlationId, payload: { leadId, fieldMap, customerId? } }`) driven by `WorkflowRunner`, with the enumerated, ordered, idempotent steps **exactly** per the architecture step table ‚Äî each with its named compensation:

   | # | Step | Forward action | Idempotency key | Compensation |
   |---|---|---|---|---|
   | 1 | `guard` | assert lead is `qualified` AND not already converted (`convertedToCustomerId == null`) | n/a (read-only) | **none** |
   | 2 | `create-customer` | create `Customer` in `prospect` from the field-map | `payload.customerId` set ‚áí skip | soft-delete the customer (`Customer.Deleted`) |
   | 3 | `link-lineage` | set `Customer.convertedFromLeadId` + `Lead.convertedToCustomerId` | both pointers set ‚áí skip | clear both pointers |
   | 4 | `lock-lead` | transition lead `qualified ‚Üí converted` (terminal/read-only) | lead already `converted` ‚áí skip | `Lead.StatusChanged` back to `qualified` |
   | 5 | `emit` | prior 4 beats already emitted `Customer.Created` + `Lead.Converted`; finalize `status=completed` | already `completed` ‚áí no-op | prior steps' compensations emit the reversing **canonical** events ‚Äî **no new event type invented** (NFR-7) |

   *Touches:* `src/features/customers/conversion/conversionSaga.ts` (step definitions), `src/shared/workflow/WorkflowRunner.ts` (engine, E0-S2) ¬∑ *Seam:* `WorkflowRunner` over `Repository<WorkflowInstance>`; each step calls the 4-beat repository mutation ¬∑ *Schema/Map:* `WorkflowInstance` type (E0-S2), `customer.types.ts` field-map.

3. **AC3 ‚Äî Resumable + idempotent.** An interrupted saga reloads from `WorkflowInstance.currentStep` and continues forward; re-running any completed step is a no-op (uses the idempotency keys in AC2: `customerId` set, both pointers set, lead already `converted`, status already `completed`).
   *Touches:* `src/shared/workflow/WorkflowRunner.ts`, `src/features/customers/conversion/conversionSaga.ts` ¬∑ *Seam:* `WorkflowRunner.resume(instanceId)` reads `currentStep`/`completedSteps` ¬∑ *Schema/Map:* idempotency keys per step.

4. **AC4 ‚Äî Explicit field-map contract (reviewed-not-extended).** The saga carries `{ name, company, contactDetails(email/phone), source, ownerId, bantNote }` from lead to customer; activity history is **linked, not duplicated** (lineage pointers, no copy). The map is locked per PRD ¬ß12 ‚Äî not extended in the pilot.
   *Touches:* `src/features/customers/conversion/fieldMap.ts`, `src/shared/domain/customer.types.ts`, `src/shared/domain/schemas.ts` ¬∑ *Seam:* Zod validate-before-persist in `create-customer` ¬∑ *Schema/Map:* `customerCreateSchema` derived from the field-map.

5. **AC5 ‚Äî Emits a linked conversion under one `correlationId` (UC-2 / NFR-7 across the saga).** The saga emits `Customer.Created` + `Lead.Converted` (each individually a conformant 4-beat mutation: one `DomainEvent` + one `AuditEvent` apiece); the single saga `correlationId` ties them together. A **linked conversion event appears on both timelines** (the lead's and the customer's Activity timeline) ‚Äî the UJ-3 climax.
   *Touches:* `src/features/customers/conversion/conversionSaga.ts`, `src/shared/events/bus.ts`, `src/shared/events/auditLog.ts`, `src/shared/events/correlation.ts` ¬∑ *Seam:* 4-beat emit/audit per step under shared `correlationId` ¬∑ *Schema/Map:* canonical events `Customer.Created`, `Lead.Converted`.

6. **AC6 ‚Äî Mid-saga failure compensates in reverse (UJ-3 edge case).** A failure at any step flips `status ‚Üí compensating` and runs compensations **in reverse from the failed step**, leaving **no half-made customer** and the lead back at `qualified`; then `status ‚Üí failed`. Compensations emit only reversing **canonical** events (`Customer.Deleted`, `Lead.StatusChanged`) ‚Äî no invented event type (NFR-7). Verified via the fault-injection toggle (ADR-007).
   *Touches:* `src/shared/workflow/WorkflowRunner.ts`, `src/features/customers/conversion/conversionSaga.ts`, `src/shared/data/faultInjection.ts` ¬∑ *Seam:* `WorkflowRunner` compensation loop ¬∑ *Schema/Map:* compensation column of AC2 table.

7. **AC7 ‚Äî UI-inspectable on a `DetailPage` variant (NFR-10).** Saga state renders on a `ConversionInspector` built as a **`DetailPage` variant ‚Äî no new layout**, showing `steps`, `currentStep`, `completedSteps`, and outcome (running/completed/compensating/failed), with four UI states via `<QueryStateBoundary>` (UC-1).
   *Touches:* `src/features/customers/conversion/ConversionInspector.tsx`, `src/shared/ui/templates/DetailPage`, `src/shared/ui/QueryStateBoundary.tsx` ¬∑ *Seam:* read `Repository<WorkflowInstance>` ¬∑ *Schema/Map:* `WorkflowInstance` (status colors only from `STATUS_TONE`).

8. **AC8 ‚Äî `ConfirmDialog` gates the Convert action (NFR-10).** The Convert action requires the shared `ConfirmDialog` before the saga starts; the mutation is optimistic with rollback on error (ADR-007), distinct toast on `409`.
   *Touches:* `src/features/customers/conversion/ConvertLeadAction.tsx`, `src/shared/ui/components/ConfirmDialog` ¬∑ *Seam:* `useMutation` (`onMutate`/`onError`/`onSettled`) ¬∑ *Schema/Map:* n/a.

## UX & Behavior (from prototype)
*Source: `prototype/app/conversion.jsx` (`ConversionInspector`), `prototype/app/store.jsx` (`commitConversion`, `getSaga/setSaga/clearSaga`). Visual oracle: `prototype/screenshots/saga-done.png`, `saga-failed.png`. Build identical to the prototype.*

- **Entry = `ConfirmDialog`** ("Convert this lead? ‚Ä¶ creates a customer in prospect, links lineage both ways, and locks the lead as converted." / confirm **"Start conversion"**) ‚Äî then hands off to the inspector (**no double-confirm**).
- **`ConversionInspector` = a `DetailPage` variant ‚Äî in-page, NOT a modal, does NOT elevate** (NFR-10). Breadcrumb **(Leads ‚Ä∫ <name> ‚Ä∫ Convert)**, back, title **"Convert <name>"**, overall status pill, and a header action.
- **Five ordered steps:** **1 Guard ¬∑ 2 Create customer ¬∑ 3 Link lineage ¬∑ 4 Lock lead ¬∑ 5 Emit.** Each renders **pending (neutral) ‚Üí current (info, spinner) ‚Üí done (success, check)**; the connector line turns green as steps complete. Each step advances one tick per **~`--crm-base` + 60ms**.
- **Right rail = field map** (Lead‚Üícustomer): Name, Company, Email, Phone, Source, Owner, BANT note + the note *"Activity history is **linked**, not copied ‚Äî the lead's timeline stays its own and the customer references it."*
- **Overall pill cycles:** **Converting‚Ä¶ (info) ‚Üí Converted (success)**, or on fault **Rolling back‚Ä¶ (warning) ‚Üí Failed (danger)**.
- **Success:** customer created in `prospect` with two-way lineage; lead locked to `converted`; one shared-`correlationId` conversion entry on **both** timelines; audit `conversion` record; lead owner notified if not the actor. Header offers **"Open customer"**; panel confirms *"Customer <name> created in prospect‚Ä¶"*; toast **"Lead converted ¬∑ Customer '<name>' created in prospect."**
- **Failure / rollback:** a **"Simulate fault"** selector (visible until done) injects a fault **at create-customer**, **at link-lineage**, or **409 at lock-lead**. On fault, phase ‚Üí `rolling-back`: completed steps **reverse top-down** (each shows **"Reversing"**, warning tone) at ~`--crm-base` + 80ms per step, landing on **Failed** with **no half-made customer** and the lead **still `qualified`**. A red panel states the rollback; header offers **"Try again"**. Toasts: generic ‚Üí persistent danger **"Conversion failed ‚Äî rolled back. No customer was created; the lead is still qualified."** (`saga-failed.png`); **409** ‚Üí distinct persistent danger **"Record changed ‚Äî conversion stopped. This lead changed while converting. Refresh and try again."**
- **Eligibility block (no saga):** entered for an ineligible lead (already converted, or not qualified) ‚Üí calm **EmptyState** **"This lead can't be converted"** + precise reason, with **"Open linked customer"** (if already converted with a link) or **"Back to lead"** ‚Äî never starts a saga. Saga runtime persists across remounts (`getSaga/setSaga/clearSaga`).
- **States:** four states via `QueryStateBoundary`; status colors only from `STATUS_TONE`.

## Inherited Universal Conformance (subset)
- **UC-1 ‚Äî Four UI states.** `ConversionInspector` and the convert flow render loading/empty/error/data via `<QueryStateBoundary>`.
- **UC-2 ‚Äî Dual events, one correlationId.** Each saga step is a 4-beat mutation (one DomainEvent + one AuditEvent); the saga `correlationId` ties all steps together (NFR-7 per-mutation rule satisfied across the saga).
- **UC-3 ‚Äî Illegal transition `422`.** `lock-lead` uses `canTransition` against `LEAD_TRANSITIONS`; a non-`qualified` source is rejected `422`.
- **UC-4 ‚Äî Conversion only from `qualified`.** Enforced by the `guard` step; re-conversion blocked.
- **UC-5 ‚Äî Tenant scope.** Lead, customer, and `WorkflowInstance` are scoped from auth context only; out-of-tenant access ‚Üí `404`.
- **TC ‚Äî Traceability.** story ‚Üí spec ‚Üí code ‚Üí test ‚Üí `Closes #`; preview green; `sprint-status.yaml`; passes `bmad-code-review`.

## Test Requirements (NFR-12)
- **Vitest:** step idempotency (each idempotency key short-circuits a re-run to a no-op); `guard` rejects `new`/`contacted`/`disqualified`/already-`converted`; resume from each `currentStep` continues forward; compensation runs in reverse from the failed step and clears all pointers + soft-deletes the customer + returns lead to `qualified`; field-map Zod validation; `correlationId` is identical across all emitted events/audits; only canonical event types emitted (no invented type).
- **RTL:** `ConversionInspector` renders four states and the step/currentStep/outcome view; `ConvertLeadAction` requires `ConfirmDialog`; optimistic rollback on error and distinct `409` toast.
- **Playwright:** happy-path conversion (qualified lead ‚Üí prospect customer, linked event on both timelines); fault-injected mid-saga failure leaves no customer and lead back at `qualified`; resume after interruption completes.
