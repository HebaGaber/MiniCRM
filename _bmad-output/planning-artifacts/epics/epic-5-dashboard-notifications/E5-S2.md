# E5-S2 — One read-model widget fed from the event log

> **GitHub Epic → Feature → Story:** Epic **E5 — Dashboard Shell, Read-Model Widget & Notifications** › Feature **5.2 — Read-Model Widget** › Story **E5-S2**
> **Cut:** Pilot · **Depends on:** E5-S1 (dashboard shell + widget slot), E0-S7 (domain-event bus + audit log)

## Story
As a tenant admin, I see one aggregate widget computed from the event log, so that I get a derived, at-a-glance roll-up without any bespoke cross-boundary write.

> **Open Question 3 — RESOLVED (DEC-UX-7): the widget is the CONVERSION FUNNEL.** The Claude Design prototype built and shipped the **conversion funnel** (`prototype/app/dashboard.jsx`; oracle `dashboard.png`); that settles OQ3. The per-subsidiary roll-up is **not** this widget — it remains the separate **E1-S5 Roll-up** surface. It still ships as exactly **ONE** widget, computed read-side from the event log (ADR-008), respecting scope. *(If Heba prefers to revert OQ3 to the roll-up, flag it — otherwise the funnel stands.)*

## Implementation-Ready Acceptance Criteria
1. **AC1 — Exactly one widget = the conversion funnel (OQ3 RESOLVED → DEC-UX-7).** `widgets/ReadModelWidget.tsx` renders **one** aggregate widget — the **conversion funnel** (five derived stages New · Contacted · Qualified · Converted · Active customers). OQ3 is **settled by the prototype** (DEC-UX-7); the per-subsidiary roll-up is **not** this widget (it remains the separate **E1-S5 Roll-up** surface). No second widget ships in the pilot.
   *Touches:* `src/features/dashboard/widgets/ReadModelWidget.tsx`, `src/features/dashboard/DashboardPage.tsx` · *Seam:* read-model projection · *Schema/Map:* aggregate read-model shape (conversion-funnel buckets).
2. **AC2 — Computed from the event log → read model (CQRS-lite, ADR-008).** The widget's aggregate is **derived from the `DomainEvent` log into a read model** — a CQRS-lite read seam. It performs **NO cross-boundary write** and is **not** event-sourcing; it is a projection for display.
   *Touches:* `src/features/dashboard/widgets/ReadModelWidget.tsx`, `src/shared/events/bus.ts` · *Seam:* event-log → read-model projection (read-only; ADR-008) · *Schema/Map:* `DomainEvent` (E0-S7) → aggregate.
3. **AC3 — Roll-up respects scope (UC-5, UJ-5 edge case).** `tenant_admin` sees the **cross-subsidiary** aggregate; a **subsidiary user sees own only** (own + parent-level `subsidiaryId=null`). Scope is taken from the auth context only; out-of-tenant data never appears (404-not-403 at the boundary).
   *Touches:* `src/features/dashboard/widgets/ReadModelWidget.tsx`, `src/shared/auth/useAuth.ts` · *Seam:* scope from auth context (ADR-002, ADR-009) · *Schema/Map:* aggregate filtered by `{tenantId, subsidiaryId}`.
4. **AC4 — Four UI states (UC-1).** The widget renders loading / empty / error / data via `<QueryStateBoundary>` — empty when no source events exist, error on projection failure.
   *Touches:* `src/features/dashboard/widgets/ReadModelWidget.tsx`, `src/shared/ui/QueryStateBoundary.tsx` · *Seam:* four-state boundary · *Schema/Map:* n/a.
5. **AC5 — Remaining widgets are explicitly later stories.** Any additional dashboard widgets are **non-goals for the pilot** and are deferred to later stories; only the single OQ3 widget is in scope here.
   *Touches:* `src/features/dashboard/widgets/` (this story adds exactly one) · *Seam:* n/a · *Schema/Map:* n/a.

## UX & Behavior (from prototype)
*Source: `prototype/app/dashboard.jsx` (`DashboardPage` conversion funnel). Visual oracle: `prototype/screenshots/dashboard.png` (New 4 · Contacted 4 · Qualified 3 · Converted 1 · Active customers 6). Build identical to the prototype. OQ3 = funnel (DEC-UX-7).*

- **Conversion funnel widget** — five derived, **read-only** stages with **horizontal bars + counts**, in order: **New · Contacted · Qualified · Converted · Active customers**. Each row = ordinal + label + proportional bar + right-aligned count.
- **Scope-aware footer line:** **"Aggregated across every subsidiary"** in tenant scope; otherwise **"Scoped to <scope> — sibling subsidiaries are not counted."**
- **Lock pill:** a **"Derived · read-only"** pill on the widget header (it is a projection, never a write).
- **Placeholder slot** beside it: *"More widgets — later stories. Only the conversion funnel ships now."* (exactly one widget ships — AC1/AC5).
- **States:** loading (bar skeletons) · **empty** (**"No activity in this scope yet"** when the scope has zero leads + customers) · error (**"Can't load dashboard data"** + retry) · ready. Re-queries on scope change via `scopeLoading`.
- **Motion:** bars skeleton→ready `--crm-base`; counts settle without travel under reduced-motion.

## Test Requirements (NFR-12)
- **Vitest:** the projection derives the aggregate purely from `DomainEvent` log entries (no write path is exercised — assert read-only); given a fixture event log, the funnel/roll-up buckets compute correctly; `tenant_admin` scope yields cross-subsidiary totals while a subsidiary user yields own-only; out-of-tenant events never contribute.
- **RTL:** `ReadModelWidget` renders all four states via `QueryStateBoundary` (empty when source log is empty, error on projection failure, data once computed); exactly one widget mounts.
- **Playwright:** on `/dashboard`, the single widget shows the scope-correct aggregate; a subsidiary user sees only own-scope numbers; feeds the cross-tenant isolation E2E.
