# E5-S1 — Role-scoped dashboard shell

> **GitHub Epic → Feature → Story:** Epic **E5 — Dashboard Shell, Read-Model Widget & Notifications** › Feature **5.1 — Dashboard Shell** › Story **E5-S1**
> **Cut:** Pilot · **Depends on:** Epic 0 kernels (E0-S9 four-state surface / `Dashboard` template, E0-S5 `AuthProvider`/scope)

## Story
As a user, I land on a dashboard scoped to my role and tenant, so that I see a relevant home surface without exposure to other tenants' or subsidiaries' data.

> **Note (PRD):** This story is **intentionally thin** — the read-model substance lives in **E5-S2**. E5-S1 ships the shell + four-state container scoped to role and tenant/subsidiary; widget content is dropped in by E5-S2.

## Implementation-Ready Acceptance Criteria
1. **AC1 — Dashboard template, role-scoped.** `DashboardPage.tsx` renders the shared `Dashboard` template and lays out role-appropriate slots (the widget slot is filled by E5-S2; notifications surface by E5-S3). Layout/visible regions derive from the actor's `roles[]` read from the auth context only.
   *Touches:* `src/features/dashboard/DashboardPage.tsx`, `src/shared/ui/templates/Dashboard` · *Seam:* `useAuth()` role/scope from auth context (ADR-009) · *Schema/Map:* n/a (no entity mutation).
2. **AC2 — Four UI states (UC-1).** The dashboard renders loading / empty / error / data through `<QueryStateBoundary>` — empty when there is nothing to show, error on read failure, data once resolved.
   *Touches:* `src/features/dashboard/DashboardPage.tsx`, `src/shared/ui/QueryStateBoundary.tsx` · *Seam:* four-state boundary (E0-S9) · *Schema/Map:* n/a.
3. **AC3 — Scoped to tenant/subsidiary (UC-5).** All data the shell reads is scoped to `{tenantId, subsidiaryId}` taken **only** from the auth context; `tenant_admin` sees tenant-level scope, a subsidiary user sees own + parent-level (`subsidiaryId=null`) only. Any out-of-tenant access resolves to `404`, never `403`.
   *Touches:* `src/features/dashboard/DashboardPage.tsx`, `src/shared/auth/useAuth.ts` · *Seam:* scope from auth context (ADR-002, ADR-009); out-of-tenant → `404` · *Schema/Map:* n/a.

- **UC-5 — Tenant scope.** Role + tenant/subsidiary scope is read from the auth context only; no cross-tenant exposure; out-of-tenant → `404`.
- **TC — Traceability.** story → spec → code → test → `Closes #`; preview green; `sprint-status.yaml`; passes `bmad-code-review`.

## Architecture — components, seams & modules touched
- **Modules (NFR-11):** `src/features/dashboard/DashboardPage.tsx`, `src/shared/ui/templates/Dashboard`, `src/shared/ui/QueryStateBoundary.tsx`, `src/shared/auth/useAuth.ts`.
- **Patterns:** four-state boundary; role-scoped layout from auth context; tenant/subsidiary scoping (404-not-403).
- **ADRs:** ADR-009 (one `AuthProvider`, claims/scope, two gates, 404-not-403), ADR-002 (`tenant_id` isolation / `subsidiary_id` in-tenant scope), ADR-006 (React Router v7 library mode + TanStack Query v5).

## Test Requirements (NFR-12)
- **Vitest:** role → visible-slot resolution (e.g. `tenant_admin` vs `support` vs `viewer`) derives only from auth-context `roles[]`; scope object is built from `{tenantId, subsidiaryId}` claims, never request input.
- **RTL:** `DashboardPage` renders all four states via `QueryStateBoundary` (loading skeleton, empty, error, data); role-conditional regions render/hide per mocked auth context.
- **Playwright:** landing on `/dashboard` shows a role-scoped shell; a user from another tenant attempting the route is scoped out (404-not-403) — feeds the cross-tenant isolation E2E.
