# E5-S1 ‚Äî Role-scoped dashboard shell

> **GitHub Epic ‚Üí Feature ‚Üí Story:** Epic **E5 ‚Äî Dashboard Shell, Read-Model Widget & Notifications** ‚Ä∫ Feature **5.1 ‚Äî Dashboard Shell** ‚Ä∫ Story **E5-S1**
> **Cut:** Pilot ¬∑ **Depends on:** Epic 0 kernels (E0-S9 four-state surface / `Dashboard` template, E0-S5 `AuthProvider`/scope)

## Story
As a user, I land on a dashboard scoped to my role and tenant, so that I see a relevant home surface without exposure to other tenants' or subsidiaries' data.

> **Note (PRD):** This story is **intentionally thin** ‚Äî the read-model substance lives in **E5-S2**. E5-S1 ships the shell + four-state container scoped to role and tenant/subsidiary; widget content is dropped in by E5-S2.

## Implementation-Ready Acceptance Criteria
1. **AC1 ‚Äî Dashboard template, role-scoped.** `DashboardPage.tsx` renders the shared `Dashboard` template and lays out role-appropriate slots (the widget slot is filled by E5-S2; notifications surface by E5-S3). Layout/visible regions derive from the actor's `roles[]` read from the auth context only.
   *Touches:* `src/features/dashboard/DashboardPage.tsx`, `src/shared/ui/templates/Dashboard` ¬∑ *Seam:* `useAuth()` role/scope from auth context (ADR-009) ¬∑ *Schema/Map:* n/a (no entity mutation).
2. **AC2 ‚Äî Four UI states (UC-1).** The dashboard renders loading / empty / error / data through `<QueryStateBoundary>` ‚Äî empty when there is nothing to show, error on read failure, data once resolved.
   *Touches:* `src/features/dashboard/DashboardPage.tsx`, `src/shared/ui/QueryStateBoundary.tsx` ¬∑ *Seam:* four-state boundary (E0-S9) ¬∑ *Schema/Map:* n/a.
3. **AC3 ‚Äî Scoped to tenant/subsidiary (UC-5).** All data the shell reads is scoped to `{tenantId, subsidiaryId}` taken **only** from the auth context; `tenant_admin` sees tenant-level scope, a subsidiary user sees own + parent-level (`subsidiaryId=null`) only. Any out-of-tenant access resolves to `404`, never `403`.
   *Touches:* `src/features/dashboard/DashboardPage.tsx`, `src/shared/auth/useAuth.ts` ¬∑ *Seam:* scope from auth context (ADR-002, ADR-009); out-of-tenant ‚Üí `404` ¬∑ *Schema/Map:* n/a.

## UX & Behavior (from prototype)
*Source: `prototype/app/dashboard.jsx` (`DashboardPage`). Visual oracle: `prototype/screenshots/dashboard.png`. Build identical to the prototype.*

- **Header:** an eyebrow reading **"<ROLE> ¬∑ <SCOPE>"** (e.g. "TENANT ADMIN ¬∑ WHOLE TENANT (ROLL-UP)"), title **"Dashboard"**, and a one-line scope-framed subtitle (admin: *"A cross-subsidiary read of the tenant. Widgets are derived and read-only."*).
- **Layout = two columns:** **left = widget column** (the E5-S2 read-model widget slot + a clearly-labeled placeholder *"More widgets ‚Äî later stories. Only the conversion funnel ships now."*); **right = Notifications card** (E5-S3 surface).
- **Re-queries on scope change** via `scopeLoading` (E1-S1) ‚Äî the shell shows its loading state then re-paints to the new scope.
- **Four states** via `QueryStateBoundary`; empty when the scope has no activity.
- *Note:* the `dashboard.png` **`PREVIEW STATE` segmented control is drift** (disabled in current code ‚Äî screens render their real store-derived state). Do **not** build it.
- **Motion:** skeleton‚Üíready `--crm-base`/`decelerate`; re-query on scope switch `--crm-base`.

## Test Requirements (NFR-12)
- **Vitest:** role ‚Üí visible-slot resolution (e.g. `tenant_admin` vs `support` vs `viewer`) derives only from auth-context `roles[]`; scope object is built from `{tenantId, subsidiaryId}` claims, never request input.
- **RTL:** `DashboardPage` renders all four states via `QueryStateBoundary` (loading skeleton, empty, error, data); role-conditional regions render/hide per mocked auth context.
- **Playwright:** landing on `/dashboard` shows a role-scoped shell; a user from another tenant attempting the route is scoped out (404-not-403) ‚Äî feeds the cross-tenant isolation E2E.
