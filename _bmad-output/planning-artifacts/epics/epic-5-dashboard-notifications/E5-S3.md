# E5-S3 — Notifications inbox/bell consuming the shared kernel

> **GitHub Epic → Feature → Story:** Epic **E5 — Dashboard Shell, Read-Model Widget & Notifications** › Feature **5.3 — In-App Notifications** › Story **E5-S3**
> **Cut:** Pilot · **Depends on:** E0-S12 (Notifications kernel), E5-S1 (dashboard shell), E4-S3 (`Ticket.Assigned`), E3-S1 (`Lead.Converted`)

## Story
As a support agent, I'm notified in-app when a ticket is assigned to me (or a lead I own converts), so that I act on work directed to me without polling.

> ## ⚠️ RE-SCOPED (ADR-014) — this story CONSUMES, it does NOT IMPLEMENT
> **Per ADR-014, Notifications is a shared kernel (E0-S12).** E5-S3 **CONSUMES** it — `NotificationsInbox.tsx` wires `useNotifications()` and renders the bell/inbox surface. The **recipient mapping** (`Ticket.Assigned → assigneeId`, `Lead.Converted → lead ownerId`) and the **event-bus subscription** **already live in `src/shared/notifications` (E0-S12)**. This story ships **NO notification logic** — no bus subscription, no recipient mapping, no bespoke writes.

## Implementation-Ready Acceptance Criteria
1. **AC1 — Driven off DomainEvents (UJ-5).** Notifications shown in the inbox/bell originate from `DomainEvent`s (e.g. `Ticket.Assigned`, `Lead.Converted`) — **not bespoke writes**. The inbox reads them via the kernel's `useNotifications()`; the bus subscription that produces them lives in E0-S12, not here.
   *Touches:* `src/features/dashboard/NotificationsInbox.tsx` (consumer), `src/shared/notifications/useNotifications.ts` (kernel, E0-S12) · *Seam:* CONSUMES Notifications kernel (ADR-014); kernel is event-driven off the bus (ADR-008) · *Schema/Map:* notification shape from E0-S12; source events `Ticket.Assigned` (E4-S3) / `Lead.Converted` (E3-S1).
2. **AC2 — Scoped to recipient's tenant/subsidiary (UC-5).** The inbox displays only the current user's notifications, already scoped to their `{tenantId, subsidiaryId}` by the kernel; no cross-tenant/subsidiary leak. Scope derives from the auth context.
   *Touches:* `src/features/dashboard/NotificationsInbox.tsx`, `src/shared/auth/useAuth.ts` · *Seam:* kernel recipient scoping (E0-S12) + auth-context scope (ADR-002, ADR-009) · *Schema/Map:* notification `{recipient, tenantId, subsidiaryId}` (E0-S12).
3. **AC3 — Consumes the shared Notifications capability (NFR-1, ADR-014).** `NotificationsInbox.tsx` **wires `useNotifications()`** and renders the **bell + inbox surface** (read/unread, mark-read). Recipient mapping + bus subscription are **NOT** re-implemented here — they live in `shared/notifications` (E0-S12). One-way layering holds: the feature consumes the shared kernel; the shared layer never imports the feature.
   *Touches:* `src/features/dashboard/NotificationsInbox.tsx` (consumer), `src/shared/notifications/NotificationService.ts` + `useNotifications.ts` (kernel — NOT modified by this story) · *Seam:* CONSUMES shared kernel; NFR-1 one-way Product→Shared · *Schema/Map:* n/a (no new logic).
4. **AC4 — Four states for the notifications surface (UC-1).** The inbox/bell renders loading / empty / error / data via `<QueryStateBoundary>` — empty when no notifications, error on read failure, data once the kernel resolves.
   *Touches:* `src/features/dashboard/NotificationsInbox.tsx`, `src/shared/ui/QueryStateBoundary.tsx` · *Seam:* four-state boundary (kernel scaffold from E0-S12) · *Schema/Map:* n/a.

## Test Requirements (NFR-12)
- **Vitest:** with the kernel emitting from a `Ticket.Assigned` event, the inbox consumer surfaces a notification to the assignee; a `Lead.Converted` event surfaces one to the lead owner; the consumer renders only the current user's tenant/subsidiary-scoped notifications (no cross-tenant leak). Assert the consumer adds **no** bus subscription or recipient mapping of its own (logic stays in E0-S12).
- **RTL:** `NotificationsInbox` renders the four states via `QueryStateBoundary`; bell badge reflects unread count; mark-read toggles read/unread state through the kernel hook.
- **Playwright:** assigning a ticket to the logged-in agent surfaces an in-app notification in the bell/inbox; a lead owned by the user converting surfaces one; another tenant's events never appear — feeds the cross-tenant isolation E2E.
