## Story
As the platform, I make the trust boundary live on the **server, not the client**, so that scoping, validation, and authorization cannot be bypassed by an untrusted client.

## Implementation-Ready Acceptance Criteria
1. **AC1 — Server-side enforcement of all three concerns.** Tenant/subsidiary scoping, Zod-equivalent validation, and the `can()` authorization predicate are enforced **server-side** in the API/service layer — the client is treated as untrusted. The server re-derives scope from the **auth context only** (never from request body params), runs the 4-beat use case (authorize → mutate → emit → audit) server-side, and rejects illegal transitions with `422`.
   *Touches:* server API/service layer (mirrors `src/shared/data/Repository.ts` contract + `src/shared/auth/permissions.ts` predicate model) · *Seam:* `Repository<T>` REST contract (NFR-5) · *Schema/Map:* server-side Zod parity of `shared/domain/schemas.ts` + `shared/domain/status.ts` transition maps.
2. **AC2 — Postgres RLS with `tenant_id`-leading indexes.** Postgres Row-Level Security policies enforce isolation in the database; **`tenant_id` leads every primary access index** (composite `tenant_id, subsidiary_id, ...`). RLS session context (`tenant_id`, `subsidiary_id`, `roles`) is set per request from the auth context. `subsidiary_id = null` ⇒ parent-level (ADR-002): `tenant_admin` sees all tenant rows; a subsidiary user sees own-subsidiary + null rows.
   *Touches:* DB migrations + RLS policies + index definitions (server) · *Seam:* Postgres RLS / session GUCs · *Schema/Map:* `tenant_id, subsidiary_id` composite indexes (ADR-001/002/003).
3. **AC3 — Cross-tenant reads return 404 (not 403).** Any read/write targeting another tenant's record returns **404** (ADR-005, NFR-6) — the record's existence is not disclosed. Out-of-tenant denial is indistinguishable from "not found"; in-tenant authorization failure is `403` and audited as `Auth.RoleDenied`.
   *Touches:* server authorization layer · *Seam:* two-gate authZ + RLS · *Schema/Map:* n/a.
4. **AC4 — Tenant-grain residency + Silo routing.** Data residency is enforced at **tenant grain** (§5, F-2). **Silo routing is retained for flagged tenants** (ADR-010): a flagged tenant's data + traffic route to its designated region/silo, configured via the flag system (OpenFeature-shaped, most-specific-wins).
   *Touches:* server routing/config + flag resolution · *Seam:* `CloudPort` / residency router · *Schema/Map:* tenant residency flag.
5. **AC5 — Production performance gate (ADR-003, HARD GATE).** p95 list query **< 50ms** with `tenant_id`-leading indexes under representative load; an **index review is a hard gate** for this story's sign-off (no merge to production without passing the index review + p95 budget).
   *Touches:* DB indexes + query plans · *Seam:* Postgres query planner / EXPLAIN review · *Schema/Map:* index review checklist.

## Test Requirements (NFR-12)
- **Vitest / server unit:** transition-map parity tests (server enforces same `LEAD/CUSTOMER/TICKET_TRANSITIONS`); Zod schema parity; `can()` predicate cell-by-cell (mirrors `permissions.test.ts`); scope-derivation-from-auth-context (rejects body-supplied tenant/subsidiary).
- **DB / RLS:** RLS policy tests proving a session scoped to tenant A cannot SELECT/UPDATE tenant B rows; `subsidiary_id = null` visibility rules (admin vs subsidiary user); index review proving `tenant_id`-leading composites; EXPLAIN/p95 benchmark asserting **< 50ms** (ADR-003 hard gate).
- **Playwright:** cross-tenant 404 assertions feed E6-S3 (the definitive isolation gate).
