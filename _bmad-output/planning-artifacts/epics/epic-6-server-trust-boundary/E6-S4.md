# E6-S4 — Transactional outbox, CQRS-lite read models, OIDC

> **GitHub Epic → Feature → Story:** Epic **E6 — Server-Side Trust Boundary** › Feature **6.4 — Production Eventing & Auth** › Story **E6-S4**
> **Cut:** DESIGN-ONLY (NOT in the 2-week pilot) · **Depends on:** E6-S1 (server) + E0-S12 (notifications kernel) + Epic 5 (read-model widgets)

> ## ⚠️ DESIGN-ONLY — NOT IN THE 2-WEEK PILOT CUT (ADR-005)
> Production target, specified as implementation-ready design. The pilot runs an **in-process event bus** and **mock SSO** behind the `AuthProvider` seam. This story graduates **eventing and auth to production shapes behind the SAME seams** — net-new server engineering, not a frontend swap.

## Story
As the platform, eventing and auth **graduate to production shapes behind the same seams**, so that the frontend (and its widgets) need no UI change while the backend becomes production-grade.

## Implementation-Ready Acceptance Criteria
1. **AC1 — In-process bus → transactional outbox + CQRS-lite read models.** The in-process bus is replaced server-side by a **transactional outbox** (events committed in the same DB transaction as the mutation) feeding **CQRS-lite read models**. Delivery is **at-least-once with idempotent consumers keyed on `eventId`** (ADR-008) — re-delivery of the same `eventId` is a no-op. Canonical event registry preserved (`PascalCaseEntity.PastTenseAction`); **not** event-sourcing.
   *Touches:* server outbox table + dispatcher + read-model projectors (production analog of `src/shared/events/bus.ts` + `auditLog.ts`) · *Seam:* event bus / outbox dispatcher · *Schema/Map:* outbox row schema + `eventId` idempotency key + canonical `eventTypes`.
2. **AC2 — correlationId preserved end-to-end.** One `correlationId` is minted at action start and threaded through mutation → outbox event → read-model projection → audit → structured log, unbroken across the async boundary (ADR-008, UC-2, NFR-7/NFR-8).
   *Touches:* server correlation propagation (production analog of `src/shared/events/correlation.ts`, `logger.ts`) · *Seam:* correlationId threading · *Schema/Map:* event/audit/log correlation field.
3. **AC3 — Mock SSO → OIDC Auth Code + PKCE behind the same `AuthProvider`.** Mock SSO is replaced by **OIDC Authorization Code + PKCE**, wired behind the **same `AuthProvider` seam** (ADR-009) — no change to feature code that consumes `useAuth`. An **IdP→Role normalization layer** maps IdP claims to the canonical role set; the claims contract `{userId, tenantId, subsidiaryId|null, roles[], exp}` is preserved; two-gate authZ and 404-not-403 unchanged.
   *Touches:* `src/shared/auth/AuthProvider.tsx` (same seam, OIDC implementation) + server OIDC + IdP→Role normalization layer · *Seam:* `AuthProvider` · *Schema/Map:* claims `{userId, tenantId, subsidiaryId|null, roles[], exp}`; `Auth.LoggedIn/LoginFailed/LoggedOut/RoleDenied` events.
4. **AC4 — Read-model widgets re-point with NO UI change.** The Epic 5 read-model/dashboard widgets **re-point to the outbox-fed read models** with **zero UI change** — only the data source behind the same query seam changes.
   *Touches:* `src/features/dashboard/widgets/*` data source (re-pointed; no UI/markup change) · *Seam:* read-model query (TanStack Query) · *Schema/Map:* read-model projection schemas.

## Test Requirements (NFR-12)

- **RTL:** dashboard widgets render unchanged against the re-pointed read models (four states preserved); `useAuth` consumers unchanged under OIDC `AuthProvider`.
- **Playwright:** OIDC Auth Code + PKCE login flow E2E; read-model widget freshness after an outbox-projected event; correlationId observable end-to-end.
