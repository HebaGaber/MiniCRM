# E2-S3 — Walk a lead through its lifecycle

> **GitHub Epic → Feature → Story:** Epic **E2 — Leads: Capture → Qualify** › Feature **2.3 — Qualification State Machine** › Story **E2-S3**
> **Cut:** Pilot · **Depends on:** E2-S1 (leads to move) + Epic 0 kernels (E0-S1 LEAD_TRANSITIONS, E0-S4 4-beat, E0-S6 guard, E0-S7 events) + E1-S1 (scope)

## Story
As a salesperson, I move a lead `new → contacted → qualified | disqualified`, with revive, so that the pipeline reflects each lead's real qualification state under enforced rules.

## Implementation-Ready Acceptance Criteria
1. **AC1 — Only LEAD_TRANSITIONS moves allowed; illegal → 422, pill unchanged (UJ-2 edge case, UC-3).** Status moves go through `POST /leads/{id}/transition` and are validated by `canTransition(lead, from, to)` against `LEAD_TRANSITIONS`. An illegal move (e.g. `new → qualified`) resolves as `422`; the lead's status is **not** mutated and the `StatusPill` stays unchanged. Legal moves: `new→[contacted,disqualified]`, `contacted→[qualified,disqualified]`, `qualified→[converted,disqualified]`.
   *Touches:* `src/features/leads/lead.service.ts`, `src/shared/domain/status.ts`, `src/shared/data/LocalStorageRepository.ts` · *Seam:* 4-beat **mutate** (transition guard) · *Schema/Map:* `LEAD_TRANSITIONS` + `canTransition`.
2. **AC2 — Revive: disqualified → contacted allowed.** `disqualified → contacted` is a legal move per the map (revive); after revive the lead is workable again and may proceed `contacted → qualified | disqualified`.
   *Touches:* `src/shared/domain/status.ts`, `src/features/leads/lead.service.ts` · *Seam:* 4-beat **mutate** · *Schema/Map:* `LEAD_TRANSITIONS` (`disqualified→[contacted]`).
3. **AC3 — Each accepted change emits Lead.StatusChanged + audit, one correlationId (UC-2).** Every accepted transition runs the full 4-beat: **authorize** (`can()` for `lead`/`transition`) → **mutate** (guard + `version` bump → `409` on mismatch) → **emit** one `Lead.StatusChanged` (carrying from/to) → **audit** one immutable `AuditEvent`, all threaded by a single `correlationId`. Uses `POST /leads/{id}/transition`, **never** `PATCH status`.
   *Touches:* `src/features/leads/lead.service.ts`, `src/shared/data/LocalStorageRepository.ts`, `src/shared/events/{bus.ts,auditLog.ts,correlation.ts}`, `src/shared/auth/permissions.ts` · *Seam:* 4-beat (authorize→mutate→emit→audit) · *Schema/Map:* canonical `Lead.StatusChanged`; `LEAD_TRANSITIONS`.
4. **AC4 — `qualified` is a manual judgment call (flag J/K).** Reaching `qualified` is an explicit human action (`contacted → qualified`); there is no automatic scoring or rule that promotes a lead. The transition is gated only by the map and the role guard, not by any computed BANT value.
   *Touches:* `src/features/leads/lead.service.ts`, `src/shared/domain/status.ts` · *Seam:* 4-beat **mutate** · *Schema/Map:* `LEAD_TRANSITIONS` (`contacted→qualified` manual).

- **UC-5 — Tenant scope.** Transition only on a lead in the caller's scope; out-of-tenant id → 404; role-gated via `can()`.

## Test Requirements (NFR-12)
- **Vitest:** `LEAD_TRANSITIONS`/`canTransition` table tests — every legal move accepted, every illegal move (incl. `new→qualified`, `new→converted`) rejected; `disqualified→contacted` revive accepted; `converted` terminal. Repository transition runs the 4-beat: emits one `Lead.StatusChanged` (from/to correct) + one `AuditEvent` sharing one `correlationId`; `version` mismatch → `409`; illegal move → `422` with status unmutated.
- **RTL:** transition control issues `POST /leads/{id}/transition`; on illegal move the `StatusPill` does not change and an error surfaces; optimistic pill update rolls back on injected fault (fault-injection toggle).
- **Playwright:** drive a lead `new → contacted → qualified`; attempt `new → qualified` and assert `422` + unchanged pill; revive a `disqualified` lead to `contacted` (`UJ-2` edge cases).
