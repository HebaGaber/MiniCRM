# E2-S2 — List and filter leads

> **GitHub Epic → Feature → Story:** Epic **E2 — Leads: Capture → Qualify** › Feature **2.2 — Lead List & Filter** › Story **E2-S2**
> **Cut:** Pilot · **Depends on:** E2-S1 (leads to list) + Epic 0 kernels (E0-S4 repository, E0-S9 ListPage/DataTable/StatusPill) + E1-S1 (tenant scope)

## Story
As a salesperson, I find leads by status and owner, so that I can work my pipeline without scrolling through every record.

## Implementation-Ready Acceptance Criteria
1. **AC1 — ListPage with DataTable + StatusPill.** The leads list renders through the shared `ListPage` template using `DataTable`; each row's status renders via `StatusPill` whose tone is read **only** from `STATUS_TONE` (no hex/px — NFR-10). Columns include name/source/owner/status.
   *Touches:* `src/features/leads/LeadsListPage.tsx`, `src/shared/ui/templates/ListPage`, `src/shared/ui/components/{DataTable,StatusPill}` · *Seam:* `Repository<T>.list` · *Schema/Map:* `STATUS_TONE` (StatusPill tone).
2. **AC2 — Filter, search, sort, pagination (Page<T>).** Filter by `status` and `ownerId`, free-text `q`, sortable columns, and pagination via `Repository<T>.list(q?) → Page<T>` (default `pageSize=25`, max 100). Filters/sort/page are passed as query params and reflected in the URL.
   *Touches:* `src/features/leads/{LeadsListPage.tsx,lead.service.ts}`, `src/shared/data/{Repository.ts,LocalStorageRepository.ts}`, `src/shared/ui/components/{FilterBar,Toolbar}` · *Seam:* `Repository<T>.list` (query → `Page<T>`) · *Schema/Map:* n/a (read path).
3. **AC3 — Scoped to tenant/subsidiary (UC-5).** The list is scoped from auth context only via the repository scope seam (ADR-002): `tenant_admin` (subsidiaryId=null) sees all tenant rows; a subsidiary user sees own + parent-level (`subsidiaryId=null`) only. No scope parameter is accepted from the caller.
   *Touches:* `src/shared/data/LocalStorageRepository.ts`, `src/features/leads/lead.service.ts` · *Seam:* `Repository<T>.list` scope filter · *Schema/Map:* n/a.
4. **AC4 — Four UI states incl. empty-with-primary-action (UC-1).** The list renders loading/empty/error/data via `<QueryStateBoundary>`; the **empty** state offers a primary "Capture a lead" action (deep-link to E2-S1), not a bare message; the error state is retryable.
   *Touches:* `src/features/leads/LeadsListPage.tsx`, `src/shared/ui/QueryStateBoundary.tsx`, `src/shared/ui/components/EmptyState` · *Seam:* four-state boundary · *Schema/Map:* n/a.

## Inherited Universal Conformance (subset)
- **UC-1 — Four UI states.** loading/empty/error/data via `<QueryStateBoundary>`; empty state carries the primary capture action.
- **UC-5 — Tenant scope.** List filtered from auth context only; out-of-scope rows never appear; out-of-tenant fetch-by-id → 404.
- **TC — Traceability.** story → spec → code → test → `Closes #`; preview green; `sprint-status.yaml`; passes `bmad-code-review`.

## Test Requirements (NFR-12)
- **Vitest:** `list` returns `Page<T>` honoring `status`/`ownerId` filters, `q` free-text, sort order, and pagination bounds (default 25, cap 100). Scope filter unit tests: admin sees all tenant rows; subsidiary user sees own + parent-level only.
- **RTL:** `LeadsListPage` renders all four states; **empty** state shows the primary "Capture a lead" action; `StatusPill` tone comes from `STATUS_TONE`; filter/sort/page controls update the query and re-render the data state.
- **Playwright:** filter by status + owner narrows the visible set; pagination navigates pages; cross-subsidiary rows are absent for a subsidiary user (`e2e/cross-tenant-isolation.spec.ts`).
