# E4-S2 — Walk a ticket through its lifecycle

> **GitHub Epic → Feature → Story:** Epic **E4 — Ticketing & Activity Timeline** › Feature **4.2 — Ticket Lifecycle & Assignment** › Story **E4-S2**
> **Cut:** Pilot · **Depends on:** E4-S1 (ticket exists in `open`) + Epic 0 kernels (E0-S1 `TICKET_TRANSITIONS`, E0-S4 repository/4-beat, E0-S6 action guard, E0-S7 dual events)

## Story
As a support agent, I move a ticket through open / in_progress / pending / resolved / closed, so that the ticket's lifecycle reflects reality and only legal moves are ever applied.

## Implementation-Ready Acceptance Criteria
1. **AC1 — Only TICKET_TRANSITIONS moves; closed terminal; reopen only resolved→open (Flag C); illegal → 422 (UC-3).** Transitions are validated against `TICKET_TRANSITIONS` via `canTransition('ticket', from, to)`: `open→[in_progress,pending,closed]`, `in_progress→[pending,resolved,open]`, `pending→[in_progress,resolved]`, `resolved→[closed,open]`, `closed→[]` (**terminal**). The **only reopen path is `resolved → open`** (Flag C). Any move outside the map is rejected **`422`** and the pill/state is unchanged.
   *Touches:* `src/shared/domain/status.ts`, `src/features/tickets/*` (transition action) · *Seam:* 4-beat authorize/validate (transition map) · *Schema/Map:* `TICKET_TRANSITIONS` + `canTransition`.
2. **AC2 — Two-step close is intentional.** Closing is a deliberate two-step path: `resolved` is **reopenable** (`resolved → open`), and only a subsequent `resolved → closed` (or `open/in_progress → closed`) reaches the terminal state — there is no direct re-open from `closed`.
   *Touches:* `src/shared/domain/status.ts`, `src/features/tickets/*` · *Seam:* transition map · *Schema/Map:* `TICKET_TRANSITIONS` (`resolved→[closed,open]`, `closed→[]`).
3. **AC3 — `pending` is the SLA-pause state — documented, no timer engine (flag D).** `pending` is documented as the SLA-pause state; **no timer/SLA engine is built** (flag D). It is reachable from `open`/`in_progress` and leaves to `in_progress`/`resolved` per the map — purely a state, no scheduled side effects.
   *Touches:* `src/shared/domain/status.ts` (doc comment), `src/features/tickets/*` · *Seam:* transition map · *Schema/Map:* `TICKET_TRANSITIONS` (`pending` edges only).
4. **AC4 — Transition via POST /tickets/{id}/transition, never PATCH status; each accepted change emits Ticket.StatusChanged + audit, one correlationId (UC-2).** State moves go through the transition endpoint `POST /tickets/{id}/transition` (REST parity, NFR-5), **never `PATCH status`**. Each accepted change runs the 4-beat and emits exactly one `Ticket.StatusChanged` DomainEvent + one AuditEvent (`before/after` status) under one `correlationId`.
   *Touches:* `src/features/tickets/*`, `src/shared/data/LocalStorageRepository.ts`, `src/shared/events/{bus.ts,auditLog.ts}` · *Seam:* 4-beat emit + audit; transition endpoint · *Schema/Map:* `TICKET_TRANSITIONS`; `Ticket.StatusChanged`.

## UX & Behavior (from prototype)
*Source: `prototype/app/tickets.jsx` (`TicketDetail`), `prototype/app/shared_records.jsx` (`ChangeStatusControl`). Build identical to the prototype.*

- **Transitions use the shared `ChangeStatusControl`** (E2-S3): legal-only next states as `→` pills; illegal states shown **locked** with an inline rule panel; **no pill change / no event** on a blocked move (422). Ticket tones: open `neutral`, in_progress `info`, pending `warning`, resolved `success`, closed `neutral`. Map matches constitution §3.2 exactly (`closed` terminal; reopen only `resolved→open`).
- **Detail header:** breadcrumb, back, title, **status pill + priority pill**, role-gated **Assign** control + **Change status**; read-only → **"Read-only"** pill.
- **`pending` = "SLA paused":** when status is `pending` the details card shows an **"SLA paused"** chip, and the transition sentence reads *"moved this ticket to pending (SLA paused)"* — documentation only, **no timer/SLA engine** (flag D).
- **Two-column body.** Left: details grid (Subject, Customer, Priority, Status, **Channel = "Email"** [C-5, decorative]) + Description + **Activity** timeline. Right: **Assignee** card ("Single assignee") + a **Customer** card linking to customer detail + **Meta**.
- **Each accepted move logs a `ticket.transition` audit record** (+ `Ticket.StatusChanged`).
- **Motion:** pill tone cross-fade `--crm-fast`; optimistic apply `--crm-instant`; rollback `--crm-slow`.

## Inherited Universal Conformance (subset)
- **UC-2 — Dual events, one correlationId.** Each accepted transition emits one `Ticket.StatusChanged` + one AuditEvent under one `correlationId`; no silent status write.
- **UC-3 — Illegal transitions rejected `422`.** Any move not in `TICKET_TRANSITIONS` (including any reopen other than `resolved → open`) is rejected `422` and never applied.
- **UC-5 — Tenant scope.** Transition acts on a ticket in the caller's scope; out-of-scope ticket id → `404`.
- **TC — Traceability.** story → spec → code → test → `Closes #`; preview green; `sprint-status.yaml`; passes `bmad-code-review`.

## Test Requirements (NFR-12)
- **Vitest (transition map):** every **legal** edge of `TICKET_TRANSITIONS` accepted; a sample of **illegal** moves rejected `422` (e.g. `open → resolved`, `closed → open`, `pending → closed`, `resolved → in_progress`); `closed` is terminal (no outgoing edges); reopen allowed only `resolved → open`. Each accepted transition emits exactly one `Ticket.StatusChanged` + one AuditEvent, same `correlationId`; rejected ones emit nothing and leave status unchanged.
- **RTL:** the detail/transition control offers only legal next states; attempting an illegal move surfaces `422` and the `StatusPill` is unchanged (`STATUS_TONE`).
- **Playwright:** a support user walks a ticket `open → in_progress → resolved → closed`, and reopens a `resolved` ticket to `open`. Contributes to SM-P5 (illegal transitions rejected) and SM-P1 (worked ticket in the thread).
