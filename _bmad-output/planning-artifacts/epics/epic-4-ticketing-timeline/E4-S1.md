# E4-S1 — Create a ticket against an eligible customer

> **GitHub Epic → Feature → Story:** Epic **E4 — Ticketing & Activity Timeline** › Feature **4.1 — Ticket Creation (customer-state gated)** › Story **E4-S1**
> **Cut:** Pilot · **Depends on:** Epic 0 kernels (E0-S1 status maps incl. `TicketStatus`, E0-S4 repository/4-beat, E0-S5 auth claims, E0-S6 action guard, E0-S7 dual events) + Epic 3 (customers exist: E3-S1 conversion, E3-S2 onboarding)

## Story
As a support agent, I open a ticket against a customer, but only when the customer is ready, so that support work only attaches to live accounts and never to a prospect, inactive, or churned record.

## Implementation-Ready Acceptance Criteria
1. **AC1 — Role gate: support/tenant_admin only; sales read-only (flag F).** Both gates apply: the route guard shows ticket-create only to `support`/`tenant_admin`; the action guard denies the create mutation for any other role. **`sales` is read-only on tickets** (flag F) — denial emits `Auth.RoleDenied` (UC-2). `viewer` denied.
   *Touches:* `src/features/tickets/*` (create surface), `src/shared/auth/permissions.ts`, `src/shared/auth/guards.tsx` · *Seam:* two-gate authZ + 4-beat authorize · *Schema/Map:* §2.2 matrix (Tickets row).
2. **AC2 — Customer-state gate as an action-guard precondition (UJ-4 edge case → 422).** Ticket creation is allowed **only when the linked customer is `active` or `onboarding`**; for `prospect`/`inactive`/`churned` the create is **blocked** as an **action-guard precondition** and rejected with **`422`** (not silently allowed) — pill/state unchanged. The gate reads `CustomerStatus` from the customer record inside the repository (scope-resolved per UC-5).
   *Touches:* `src/features/tickets/*` (create use case), `src/shared/domain/customer.types.ts` (read `CustomerStatus`), `src/shared/auth/permissions.ts` (precondition) · *Seam:* 4-beat authorize beat (precondition guard) · *Schema/Map:* eligible-state set `{active, onboarding}`; `422` on violation.
3. **AC3 — Links to exactly one Customer; inherits tenant/subsidiary.** A ticket carries exactly one `customerId` and **inherits the customer's `tenantId`/`subsidiaryId`** — the ticket is created in the customer's scope (never a caller-supplied scope, UC-5); out-of-scope `customerId` resolves not-found (`404`).
   *Touches:* `src/shared/domain/ticket.types.ts`, `src/shared/data/LocalStorageRepository.ts` · *Seam:* `Repository<T>.create` scope resolution · *Schema/Map:* `Ticket` Zod schema (`customerId` required, single).
4. **AC4 — Priority vocabulary; severity is not a field.** `priority ∈ low | medium | high | urgent` (`TicketPriority` literal). **Severity is vocabulary only — not a field** on `Ticket`.
   *Touches:* `src/shared/domain/ticket.types.ts`, `src/shared/domain/schemas.ts` · *Seam:* Zod validate-before-persist · *Schema/Map:* `TicketPriority` literal in `Ticket` Zod schema.
5. **AC5 — Created in `open`; emits Ticket.Created + audit (UC-2).** A valid create produces a `Ticket` in initial status **`open`** and, via the 4-beat, emits exactly one `Ticket.Created` DomainEvent + one AuditEvent sharing one `correlationId`. Invalid input → `422` + field details.
   *Touches:* `src/features/tickets/*`, `src/shared/data/LocalStorageRepository.ts`, `src/shared/events/{bus.ts,auditLog.ts}` · *Seam:* 4-beat emit + audit · *Schema/Map:* `Ticket` Zod schema; initial `open`.
6. **AC6 — Four ticket states defined (UC-1 vocabulary).** The four-state vocabulary `open / in_progress / pending / resolved / closed` (`TicketStatus`) is the single source from `shared/domain/status.ts`; the create surface renders the four UI states via `<QueryStateBoundary>`.
   *Touches:* `src/shared/domain/status.ts`, `src/features/tickets/*`, `src/shared/ui/QueryStateBoundary.tsx` · *Seam:* four-state boundary · *Schema/Map:* `TicketStatus` literal (state vocabulary owned here, transitions in E4-S2).

## Inherited Universal Conformance (subset)
- **UC-1 — Four UI states.** The ticket-create surface (`EntityForm` variant) renders loading/empty/error/data via `<QueryStateBoundary>`.
- **UC-2 — Dual events, one correlationId.** The create emits exactly one `Ticket.Created` + one AuditEvent under one `correlationId`; a denied or precondition-blocked attempt emits `Auth.RoleDenied` rather than a silent write.
- **UC-5 — Tenant scope.** Ticket inherits the customer's tenant/subsidiary; scope read from context only; out-of-scope `customerId` → `404`.
- **TC — Traceability.** story → spec → code → test → `Closes #`; preview green; `sprint-status.yaml`; passes `bmad-code-review`.
