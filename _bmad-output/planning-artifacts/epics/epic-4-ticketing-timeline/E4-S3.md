# E4-S3 ‚Äî Assign a ticket to a single assignee

> **GitHub Epic ‚Üí Feature ‚Üí Story:** Epic **E4 ‚Äî Ticketing & Activity Timeline** ‚Ä∫ Feature **4.2 ‚Äî Ticket Lifecycle & Assignment** ‚Ä∫ Story **E4-S3**
> **Cut:** Pilot ¬∑ **Depends on:** E4-S1 (ticket exists) + **E0-S12 (Notifications kernel ‚Äî CONSUMED)** + Epic 0 kernels (E0-S4 repository/4-beat, E0-S6 action guard, E0-S7 dual events)

## Story
As a support agent, I assign a ticket to one support user, so that exactly one person owns the ticket and is notified to pick it up.

## Implementation-Ready Acceptance Criteria
1. **AC1 ‚Äî Single assigneeId (a support user).** A ticket carries **exactly one `assigneeId`**, which must reference a `support` user in the ticket's tenant/subsidiary scope. Reassignment replaces the single `assigneeId` (no multi-assignee). An out-of-scope or non-`support` assignee is rejected (`422`).
   *Touches:* `src/shared/domain/ticket.types.ts`, `src/features/tickets/*` (assign action), `src/shared/domain/schemas.ts` ¬∑ *Seam:* `Repository<T>.update` + 4-beat validate ¬∑ *Schema/Map:* `Ticket` Zod (`assigneeId` single, optional‚Üíset; role check `support`).
2. **AC2 ‚Äî Emits Ticket.Assigned + audit, one correlationId (UC-2).** An accepted assignment runs the 4-beat and emits exactly one `Ticket.Assigned` DomainEvent + one AuditEvent (`before/after` assignee) under one `correlationId`.
   *Touches:* `src/features/tickets/*`, `src/shared/data/LocalStorageRepository.ts`, `src/shared/events/{bus.ts,auditLog.ts}` ¬∑ *Seam:* 4-beat emit + audit ¬∑ *Schema/Map:* `Ticket.Assigned` canonical event.
3. **AC3 ‚Äî Assignment triggers the in-app notification by CONSUMING the E0-S12 Notifications kernel (architecture delta).** The in-app notification is produced **by the `NotificationService` (E0-S12) subscribing to the bus** and mapping **`Ticket.Assigned ‚Üí assigneeId`**, scoped to the assignee's tenant/subsidiary. Tickets **do not write notifications directly** ‚Äî the notification is event-driven off `Ticket.Assigned` only. This consumes the kernel per ADR-014.
   *Touches:* `src/shared/notifications/NotificationService.ts` (`Ticket.Assigned ‚Üí assigneeId` handler), `src/features/tickets/*` (emits the event only) ¬∑ *Seam:* Notifications kernel subscribing to the DomainEvent bus ¬∑ *Schema/Map:* event-type ‚Üí recipient map (`Ticket.Assigned ‚Üí assigneeId`).
4. **AC4 ‚Äî Group/queue routing out of scope (extension point).** Routing to a group/queue is **out of scope**; the single-`assigneeId` mapping is documented as the extension point where queue routing would later hook in.
   *Touches:* `src/features/tickets/*` (doc/extension note), `src/shared/notifications/NotificationService.ts` ¬∑ *Seam:* extension point ¬∑ *Schema/Map:* n/a.

## Inherited Universal Conformance (subset)
- **UC-2 ‚Äî Dual events, one correlationId.** Assignment emits one `Ticket.Assigned` + one AuditEvent under one `correlationId`; the notification is a downstream subscriber, not a second write path.
- **UC-5 ‚Äî Tenant scope.** The assignee must be in scope; the notification is scoped to the assignee's tenant/subsidiary ‚Äî no cross-tenant notification leak. Out-of-scope ticket id ‚Üí `404`.
- **TC ‚Äî Traceability.** story ‚Üí spec ‚Üí code ‚Üí test ‚Üí `Closes #`; preview green; `sprint-status.yaml`; passes `bmad-code-review`.

## Test Requirements (NFR-12)
- **Vitest:** assignment sets a single `assigneeId`; a non-`support` or out-of-scope assignee ‚Üí `422`; an accepted assignment emits exactly one `Ticket.Assigned` + one AuditEvent, same `correlationId`; **`NotificationService` (E0-S12), subscribed to the bus, produces one notification to `assigneeId`** on `Ticket.Assigned`, scoped to the assignee's tenant/subsidiary (no cross-tenant leak); no notification is produced by any path other than the domain event.
- **RTL:** the assign control on the ticket detail accepts one assignee; reassign replaces it; the assigned user's notifications surface (via `useNotifications()`, E0-S12) reflects the new notification.
- **Playwright:** a support agent assigns a ticket and the assignee sees the in-app notification ‚Äî feeding the Epic 5 inbox (E5-S3, which also consumes E0-S12). Contributes to SM-P1 (worked ticket) and SM-P4 (one audit + one domain event).
