# E4-S4 — Unified per-record activity timeline

> **GitHub Epic → Feature → Story:** Epic **E4 — Ticketing & Activity Timeline** › Feature **4.3 — Activity Timeline (audit-as-feature)** › Story **E4-S4**
> **Cut:** Pilot · **Depends on:** E4-S1/S2/S3 (tickets emit `Ticket.Created/StatusChanged/Assigned`), E3 conversion lineage (`Lead.Converted`/`Customer.Created`) + Epic 0 kernels (E0-S6 permissions incl. `own`, E0-S7 dual events + audit log)

## Story
As any scoped user, I read one record's full story — lineage, status changes, assignments — in time order, so that the record's history is legible to anyone permitted to see the record, while raw compliance detail stays behind the audit matrix.

## Implementation-Ready Acceptance Criteria
1. **AC1 — ActivityTimeline renders the record's DomainEvents chronologically, gated by permission to VIEW THE RECORD (ADR-016).** `ActivityTimeline.tsx` renders the `DomainEvent` stream **filtered by `entityType + entityId`**, ordered chronologically — the per-record **product** surface. It is gated **by permission to view that record** (not by the audit matrix), so a `viewer` within scope **sees it**.
   *Touches:* `src/features/tickets/ActivityTimeline.tsx`, `src/shared/events/bus.ts` (DomainEvent read), `src/shared/auth/guards.tsx` (record-view gate) · *Seam:* DomainEvent stream filtered by `entityType+entityId`; record-view guard · *Schema/Map:* `DomainEvent` shape; n/a transition.
2. **AC2 — On a customer it INTERLEAVES conversion lineage with the ticket lifecycle (UJ-4 climax).** On a customer record the timeline **interleaves the conversion lineage** (`Lead.Converted`, `Customer.Created` from E3) **with the ticket lifecycle** (`Ticket.Created/StatusChanged/Assigned` from E4-S1/S2/S3) in a single chronological stream — the UJ-4 climax. Linked, not duplicated.
   *Touches:* `src/features/tickets/ActivityTimeline.tsx`, `src/shared/events/bus.ts` · *Seam:* DomainEvent stream (multi-`entityType` interleave by `correlationId`/timestamp) · *Schema/Map:* `DomainEvent` shape.
3. **AC3 — Audit/Events log is a SEPARATE, raw compliance view gated by the §2.2 matrix (ADR-016, so the matrix is not violated).** `AuditLogPanel.tsx` is a **separate** view showing the **raw `AuditEvent` `before/after`**, gated by the §2.2 **"View audit / events"** row via the E0-S6 guard / ADR-015 predicate: `tenant_admin` = **all**, `sales`/`support` = **`own`** (the ADR-015 `own` predicate: `ownerId`/`assigneeId`/`actorId === userId`), **`viewer` = none**. Denial emits nothing to `viewer`; the matrix is never bypassed by the timeline.
   *Touches:* `src/features/tickets/AuditLogPanel.tsx`, `src/shared/events/auditLog.ts` (AuditEvent read), `src/shared/auth/permissions.ts` (`own` predicate) · *Seam:* raw AuditEvent read + matrix-`own` guard (E0-S6) · *Schema/Map:* `AuditEvent` (`before/after`); §2.2 "View audit/events" row.
4. **AC4 — Four UI states (UC-1).** Both surfaces render loading/empty/error/data via `<QueryStateBoundary>`; a record with no events shows the empty state (not an error); a `viewer` on `AuditLogPanel` sees the gated/empty (none) state.
   *Touches:* `src/features/tickets/{ActivityTimeline.tsx,AuditLogPanel.tsx}`, `src/shared/ui/QueryStateBoundary.tsx` · *Seam:* four-state boundary · *Schema/Map:* n/a.

## UX & Behavior (from prototype)
*Source: `prototype/app/shared_records.jsx` (`RecordTimeline`), `prototype/app/audit.jsx` (`AuditLog`, `Store.auditFor`, `ACTION_META`). Visual oracle: `prototype/screenshots/audit.png` (+ `audit2..4.png`). Build identical to the prototype — these are TWO distinct surfaces (DEC-UX-5 / ADR-016).*

- **Activity timeline (`RecordTimeline`)** — the per-record **product** surface. Newest-first vertical timeline; each entry = **bold actor + human sentence + relative time** (absolute on hover) + connector line. **Kind tags** distinguish `conversion` (**"Lineage"**, green) and `ticket` (**"Ticket"**, info) entries from plain status entries. On a **customer** it **interleaves** conversion lineage + status changes + ticket lifecycle in one stream (UJ-4 climax). Record-view gated (**viewer included within scope**). Empty state per record type; new entries fade in at `--crm-fast`.
- **Audit/Events log (`AuditLog`)** — a **separate top-level** raw, immutable, before/after **compliance** table (nav "Audit & events"; hidden entirely for viewer):
  - **Dark "Raw audit log" banner** stating the matrix gate — *"All records across the tenant"* (admin) / *"Only records you acted on"* (agents) — + a **"Compliance"** tag.
  - **Columns:** **Timestamp** (absolute, tabular) · **Actor** · **Action** (tone-coded pill with icon) · **Record & change** (record label + `before`→`after` as code chips: **before neutral, after green-soft**) · **Scope**.
  - **Filters:** Action + Entity dropdowns (derived from visible rows) + a live **"N records"** count.
  - **Matrix gate (`auditFor`):** tenant_admin → all rows; sales/support → only rows where they are the `actor` (`own` predicate); **viewer → none**.
  - **Action taxonomy (`ACTION_META`):** `permission.change` (warning) · `lead.create`/`ticket.create` (neutral) · `lead.transition`/`customer.transition`/`ticket.transition` (info, `repeat`) · `ticket.assign` (info) · `subsidiary.offboard` (danger) · `conversion` (success, `git-merge`).
  - **States:** loading (row skeletons) · empty (admin "No audit records yet" / agent "No records you acted on") · error (retry) · ready.

## Test Requirements (NFR-12)
- **Vitest:** `ActivityTimeline` returns the `DomainEvent` stream filtered by `entityType+entityId` in chronological order; on a customer, conversion-lineage events (`Lead.Converted`, `Customer.Created`) interleave with ticket events (`Ticket.Created/StatusChanged/Assigned`) by timestamp; `AuditLogPanel` exposes raw `AuditEvent` `before/after` only to permitted roles — `tenant_admin` all, `sales`/`support` `own` (via the `own` predicate), `viewer` none; the record-view gate admits a `viewer` to `ActivityTimeline` but the matrix gate denies the same `viewer` `AuditLogPanel` (matrix not violated).
- **RTL:** both panels render the four states via `<QueryStateBoundary>` (empty timeline → empty state; `viewer` on `AuditLogPanel` → gated/empty); the two surfaces are distinct components.
- **Playwright:** UJ-4 climax — on a converted-then-ticketed customer, a scoped user reads one interleaved timeline; a `viewer` sees the activity timeline but not the audit log. Contributes to SM-P1 (end-to-end thread legible) and SM-P4 (events/audit assertable).
