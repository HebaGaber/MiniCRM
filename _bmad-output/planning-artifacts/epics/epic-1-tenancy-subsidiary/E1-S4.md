# E1-S4 — Switcher in AppShell

> **GitHub Epic → Feature → Story:** Epic **E1 — Tenancy & Subsidiary** › Feature **1.3 — Tenant/Subsidiary Switcher** › Story **E1-S4**
> **Cut:** Pilot · **Depends on:** E1-S1 (scope seam), E1-S2 (subsidiary list)

## Story
As a tenant admin, I switch active subsidiary scope, so that the data visibly changes to the unit I am working in (UJ-1 climax).

## Implementation-Ready Acceptance Criteria
1. **AC1 — Switcher in AppShell.** A subsidiary switcher control lives in the shared `AppShell`; it lists the admin's same-tenant active subsidiaries plus a "whole tenant / parent-level" option.
   *Touches:* `src/shared/ui/components/AppShell` · *Seam:* AppShell scope control · *Schema/Map:* n/a.
2. **AC2 — Selecting sets active subsidiary scope in auth context (validated, not trusted).** Selection updates the **active subsidiary scope** in the auth context; the pilot adapter **validates the chosen subsidiary belongs to the token's tenant**, mirroring the production `X-Subsidiary-Id` header contract (constitution §5.2, NFR-6). The scope value is **never trusted as a raw client value** for cross-tenant authorization — `tenantId` always comes from the token.
   *Touches:* `src/shared/auth/{AuthProvider.tsx,useAuth.ts}`, `src/app/providers.tsx` · *Seam:* auth-context scope mutation + tenant-membership validation · *Schema/Map:* claims (ADR-009).
3. **AC3 — Visible data set changes to match (UJ-1 climax).** On switch, the active scope feeds `TenantContext` → repository scope resolution (E1-S1), so lists/reads re-query and the visible data set changes; TanStack Query caches are keyed by scope so stale cross-scope data is not shown.
   *Touches:* `src/app/providers.tsx`, `src/shared/data/LocalStorageRepository.ts`, query keys in `src/features/*` · *Seam:* `TenantContext` → `Repository<T>` scope · *Schema/Map:* n/a.
4. **AC4 — Non-admins see only their own scope.** A subsidiary user has no switcher choices beyond their own scope (the control is read-only/absent); they cannot select another subsidiary or parent-level.
   *Touches:* `src/shared/ui/components/AppShell`, `src/shared/auth/{guards.tsx,permissions.ts}` · *Seam:* two-gate authZ on switcher options · *Schema/Map:* n/a.

## UX & Behavior (from prototype)
*Source: `prototype/app/shell.jsx` (`ScopeSwitcher`), `prototype/app/store.jsx` (`setScope`). Build identical to the prototype.*

- **Chip** shows the tenant name over the current scope name. For **scope-fixed roles** (sales/support/viewer) the chip is **disabled** with a `lock` icon, title **"Scope is fixed for your role"**. For admin it shows `chevrons-up-down` and opens a dropdown.
- **Dropdown options:** **"Whole tenant (roll-up)"** (`layers` icon) first, then each **active** subsidiary (`building-2`, region as subline). The active option carries a **check**. Enter with `crm-pop` at `--crm-base`/`decelerate`.
- **On switch:** `setScope` → **`scopeLoading` skeleton re-query** (E1-S1) → the visible data set changes to the chosen scope (query caches keyed by scope so no stale cross-scope data).
- **Offboarded subsidiaries disappear** from the switcher (`activeSubs`); if the current scope is offboarded, scope **snaps back to tenant**.
- **Motion:** dropdown enter `--crm-base`/`decelerate`; the post-switch re-query skeleton→data cross-fade `--crm-base`.
- *Visual oracle:* the locked chip (non-admin) and the open dropdown (admin) appear across the shell screenshots; `dashboard.png` shows "Whole tenant (roll-up)".

### Correct-course fidelity pins — UI source of truth (DEC-CC-5, 2026-06-08)
*Prototype `prototype/app/shell.jsx` (`ScopeSwitcher`, `Sidebar`) + `config.jsx` (`NAV`) + `store.jsx` (`setScope`) is the oracle.*
- **Sidebar nav grouping (prototype wins):** two labelled groups — **Workspace** (Dashboard, Leads, Customers, Tickets) and **Tenancy** (Subsidiaries, Roll-up, Audit & events) — not one flat list. (Dashboard/Audit are other epics' surfaces, but the **group labels + ordering + Subsidiaries/Roll-up placement under "Tenancy"** are pinned here.)
- **Per-item icons (prototype wins — currently swapped):** **Subsidiaries = `network`**, **Roll-up = `layers`**. The impl uses `layers` for Subsidiaries and `bar-chart-2` for Roll-up — both wrong.
- **Roll-up nav role-gating (DEC-CC-6 — spec wins, prototype stale):** the Roll-up entry is visible to **every role**, gated by **`rollup.view`** (admin tenant-wide; sales/support/viewer scoped) — **not** the prototype's `roles: ['tenant_admin']`. This is the one place the prototype nav config (`config.jsx:43`) is the stale artifact; the rest of the nav follows the prototype. (The implementation is already correct here.)
- **Switcher dropdown copy/icons (pin — impl matches, keep it):** header **"Northwind Trading · scope"**; first option **"Whole tenant (roll-up)"** (`layers` icon) with subline **"Aggregate across the tenant"** and a check on the active option; divider; then each **active** subsidiary (`building-2`, region as subline). Locked chip (scope-fixed roles) shows the `lock` icon, title **"Scope is fixed for your role"**.
- **Post-switch re-query skeleton (§8.6):** `setScope`/`setSubsidiaryScope` drives the **`scopeLoading`** skeleton→data cross-fade keyed off a **named constant derived from `--crm-base`** (the prototype's `base + 220ms`), with TanStack Query caches **keyed by scope** so no stale cross-scope data shows. Verify this is wired in the shell, not just at the repository.
- **Offboarded-scope snap-back:** shared with E1-S3 — if the active scope is offboarded it snaps back to tenant (`setSubsidiaryScope(null, …)`); offboarded subs disappear from `activeSubs`.

## Inherited Universal Conformance (subset)

- **UC-5 — Tenant scope.** Switch only re-scopes **within** the token's tenant; an out-of-tenant scope value is rejected by the adapter (never trusted), preserving 404-not-403 semantics for foreign data.

## Test Requirements (NFR-12)
- **Vitest:** adapter validation — a chosen subsidiary not belonging to the token's tenant is rejected; `tenantId` is never read from the chosen scope; scope change updates `TenantContext`. Permission cell — subsidiary user gets only their own scope as an option.
- **RTL:** switcher renders same-tenant active subsidiaries for admin; absent/read-only for non-admin; switching re-queries and the rendered data set changes; empty scope → empty state.
- **Playwright:** **UJ-1 climax** — admin switches subsidiary and the visible list changes to that subsidiary's data. **cross-tenant isolation** — a forged/foreign subsidiary scope value is rejected (no cross-tenant data appears, 404 semantics preserved); non-admin has no switcher choices.
