# E1-S3 — Offboard a subsidiary (soft-delete + reassign orphans)

> **GitHub Epic → Feature → Story:** Epic **E1 — Tenancy & Subsidiary** › Feature **1.2 — Subsidiary Onboard / Offboard** › Story **E1-S3**
> **Cut:** Pilot · **Depends on:** E1-S1, E1-S2; Epic 0 kernels (E0-S7 events/audit, ConfirmDialog)

## Story
As a tenant admin, I offboard a subsidiary without orphaning or hard-deleting its records, so that the unit closes cleanly and its data stays owned and visible.

## Implementation-Ready Acceptance Criteria
1. **AC1 — tenant_admin-only.** Route + action guarded by `can(actor, 'subsidiary', 'softDelete', resource)`; non-admin denied (404 out-of-tenant / 403 in-tenant, audited `Auth.RoleDenied`). Hard-delete is never granted to anyone.
   *Touches:* `src/app/router.tsx`, `src/shared/auth/{guards.tsx,permissions.ts}` · *Seam:* two-gate authZ + 4-beat authorize · *Schema/Map:* n/a.
2. **AC2 — Soft-delete the subtree, never hard-delete (flag I).** Offboard sets `deletedAt` on the subsidiary (and subtree); records are never physically removed. Soft-delete flows through `Repository<T>.remove`.
   *Touches:* `src/shared/data/LocalStorageRepository.ts`, `src/shared/domain/tenant.types.ts` · *Seam:* `Repository<T>.remove` (soft, 4-beat) · *Schema/Map:* n/a.
3. **AC3 — Reassignment rule (before commit).** In the offboard dialog the admin chooses a **target** — either another **active** subsidiary in the same tenant, or **parent level (`subsidiaryId=null`)**. Before the subsidiary soft-delete commits, every **active (`deletedAt=null`)** lead/customer/ticket owned by the subsidiary is reassigned: set `subsidiaryId` to the target. **Owner re-scope (DEC-CC-8, 2026-06-08 ruling):** the record's existing `ownerId`/`assigneeId` is **preserved** — moved untouched with the record — and is **NOT** reassigned to the acting admin/`session.userId`. (The prototype re-scopes owners to the target subsidiary's `SUB_PEOPLE` person, but that static roster has no equivalent in this app — subsidiary IDs are runtime-generated, only the mock identities exist, and `Customer` has no owner field — so the build-safe resolution is to preserve the existing owner; only `subsidiaryId` moves.) Each reassigned record emits its own `<Entity>.Updated` + audit **under the offboard correlationId**.
   *Touches:* `src/features/{leads,customers,tickets}`, `src/shared/data/LocalStorageRepository.ts`, `src/shared/events/{bus.ts,auditLog.ts,correlation.ts}`, `src/shared/ui/components/ConfirmDialog` · *Seam:* batched `Repository<T>.update` (4-beat per record) + correlationId · *Schema/Map:* `Lead.Updated`/`Customer.Updated`/`Ticket.Updated` (canonical).
4. **AC4 — Emits Tenant.SubsidiaryRemoved + audit, sharing the correlationId (UC-2).** The subsidiary soft-delete emits one `Tenant.SubsidiaryRemoved` `DomainEvent` + one `AuditEvent`, both threaded with the **same** offboard correlationId as all reassignment events.
   *Touches:* `src/shared/events/{bus.ts,auditLog.ts,eventTypes.ts,correlation.ts}` · *Seam:* 4-beat emit + audit · *Schema/Map:* `Tenant.SubsidiaryRemoved` (canonical).
5. **AC5 — ConfirmDialog required (destructive).** The destructive offboard requires explicit confirmation via the shared `ConfirmDialog`, which also surfaces the reassignment-target selector before commit.
   *Touches:* `src/shared/ui/components/ConfirmDialog` · *Seam:* destructive-action confirmation · *Schema/Map:* n/a.
6. **AC6 — Soft-deleted excluded from lists unless includeDeleted=true.** Per constitution §4, lists exclude soft-deleted rows unless `filter.includeDeleted=true`; the offboarded subsidiary disappears from the switcher and lists by default.
   *Touches:* `src/shared/data/LocalStorageRepository.ts` · *Seam:* `Repository<T>.list` filter · *Schema/Map:* `Page<T>`/list filter.

## UX & Behavior (from prototype)
*Source: `prototype/app/tenancy.jsx` (`OffboardDialog`), `prototype/app/store.jsx` (`commitOffboard`). The marquee tenancy edge flow — build identical to the prototype.*

- **`OffboardDialog` = danger `ConfirmDialog` with a batch-reassign saga.** Phases `choose → running`.
- **Impact preview:** counts of **active** records that must move — **Leads / Customers / Tickets** (only non-terminal records count; `ACTIVE_STATES` excludes converted/churned/closed).
- **Required field "Reassign active records to":** another **active** subsidiary in the same tenant **or "Parent level (shared)"**. Plus a **"Simulate a mid-batch failure"** toggle.
- **Run (success):** progress bar ticks **`done / total`** at ~`max(90, base/2)` per record. On commit → soft-deletes the sub, reassigns each active record's `subsidiaryId` to the target and re-scopes its `owner`/`assignee` to the target's default person; **success toast** *"<sub> offboarded. N active records reassigned to <target>."* If the current scope was the offboarded sub, scope resets to tenant.
- **Run (mid-batch fault):** at **~60%** the run aborts, **nothing is committed**, a **persistent danger toast** fires — *"Reassignment failed mid-batch — rolled back. No records moved. <sub> is still active."* — with **Try again**, and the dialog closes.
- **ConfirmDialog discipline:** danger-toned; initial focus on the **safe Cancel** control (never the destructive button); **`Esc` is suppressed while running**.
- **Motion:** dialog enter `--crm-base`; progress ticks ~`max(90, base/2)`/record; failed-run rollback is the loud reversal (toast `--crm-fast`, persistent).

### Correct-course fidelity pins — UI source of truth (DEC-CC-5, 2026-06-08)
*Prototype `prototype/app/tenancy.jsx` (`OffboardDialog`) + `store.jsx` (`commitOffboard`) is the oracle; these pin the audit findings.*
- **Owner re-scope (behavioral — see AC3, DEC-CC-8):** orphaned `ownerId`/`assigneeId` are **preserved** (moved untouched with the record); the offboard saga reassigns only `subsidiaryId` and must **not** assign records to the acting admin. (Resolved away from the prototype's `SUB_PEOPLE` re-scope: no per-subsidiary roster exists in the app.)
- **Scope snap-back (behavioral):** if the offboarded subsidiary **is the current active scope**, reset it via **`setSubsidiaryScope(null, tenantId)`** (snap back to tenant) as part of finishing the offboard — do not rely solely on the switcher dropping the entry, since `session.subsidiaryId` would otherwise still point at the dead sub.
- **Tick cadence (§8.6/NFR-10):** the per-record step is a **named constant derived from `--crm-base`** (`max(90, base/2)`), read from the token at runtime — **not** a hardcoded `100`/`200` ms literal in the component.
- **Impact card background:** token **`--iso-blue-3-50`** (not `--iso-brand-soft`/`--iso-blue-3-100`).
- **Modal scrim (DEC-CC-4 / §8.6):** the DS scrim+blur tokens (`--crm-scrim` + `--crm-backdrop-blur`), not a raw `rgba()`.
- **Exact strings:** fault toast *"Reassignment failed mid-batch — rolled back. No records moved. `<sub>` is still active."* (persistent, **Try again**); success toast *"`<sub>` offboarded. N active records reassigned to `<target>`."* (or *"No active records to reassign."*). Initial focus on the **safe Cancel**; **`Esc` suppressed while running**.

## Inherited Universal Conformance (subset)
- **UC-1 — Four UI states.** Offboard dialog + post-offboard lists render all four states; the batch operation shows progress/error and rolls back optimistically on failure.
- **UC-2 — Dual events, one correlationId.** Every reassignment `<Entity>.Updated` and the final `Tenant.SubsidiaryRemoved` share one offboard correlationId, each paired with its audit event.
- **UC-5 — Tenant scope.** Reassignment targets are constrained to the **same tenant** (another active subsidiary or parent-level); cross-tenant reassignment is impossible (scope from context).
- **TC — Traceability.** story → spec → code → test → `Closes #`; preview green; `sprint-status.yaml`; passes `bmad-code-review`.

## Test Requirements (NFR-12)
- **Vitest:** soft-delete sets `deletedAt` (never hard-deletes); reassignment moves only **active** records, sets `subsidiaryId` to target and re-scopes `ownerId`/`assigneeId`; one correlationId spans all reassignment `<Entity>.Updated` + the `Tenant.SubsidiaryRemoved`; target validation rejects an inactive/other-tenant subsidiary; permission cell — only tenant_admin may offboard; hard-delete never granted.
- **RTL:** ConfirmDialog blocks until confirmed and requires a target selection; four states on the batch; optimistic rollback when a reassignment fails mid-batch.
- **Playwright:** offboard happy path (admin picks target → records reassigned, subsidiary leaves switcher/lists, reappears with `includeDeleted=true`); **cross-tenant isolation** — reassignment target picker offers only same-tenant active subsidiaries + parent-level, never another tenant's subsidiaries; offboarded data never leaks cross-tenant.
