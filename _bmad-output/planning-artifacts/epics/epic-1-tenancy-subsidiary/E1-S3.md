# E1-S3 ‚Äî Offboard a subsidiary (soft-delete + reassign orphans)

> **GitHub Epic ‚Üí Feature ‚Üí Story:** Epic **E1 ‚Äî Tenancy & Subsidiary** ‚Ä∫ Feature **1.2 ‚Äî Subsidiary Onboard / Offboard** ‚Ä∫ Story **E1-S3**
> **Cut:** Pilot ¬∑ **Depends on:** E1-S1, E1-S2; Epic 0 kernels (E0-S7 events/audit, ConfirmDialog)

## Story
As a tenant admin, I offboard a subsidiary without orphaning or hard-deleting its records, so that the unit closes cleanly and its data stays owned and visible.

## Implementation-Ready Acceptance Criteria
1. **AC1 ‚Äî tenant_admin-only.** Route + action guarded by `can(actor, 'subsidiary', 'softDelete', resource)`; non-admin denied (404 out-of-tenant / 403 in-tenant, audited `Auth.RoleDenied`). Hard-delete is never granted to anyone.
   *Touches:* `src/app/router.tsx`, `src/shared/auth/{guards.tsx,permissions.ts}` ¬∑ *Seam:* two-gate authZ + 4-beat authorize ¬∑ *Schema/Map:* n/a.
2. **AC2 ‚Äî Soft-delete the subtree, never hard-delete (flag I).** Offboard sets `deletedAt` on the subsidiary (and subtree); records are never physically removed. Soft-delete flows through `Repository<T>.remove`.
   *Touches:* `src/shared/data/LocalStorageRepository.ts`, `src/shared/domain/tenant.types.ts` ¬∑ *Seam:* `Repository<T>.remove` (soft, 4-beat) ¬∑ *Schema/Map:* n/a.
3. **AC3 ‚Äî Reassignment rule (before commit).** In the offboard dialog the admin chooses a **target** ‚Äî either another **active** subsidiary in the same tenant, or **parent level (`subsidiaryId=null`)**. Before the subsidiary soft-delete commits, every **active (`deletedAt=null`)** lead/customer/ticket owned by the subsidiary is reassigned: set `subsidiaryId` to the target, and re-scope `ownerId`/`assigneeId` to the target where the prior owner is being offboarded. Each reassigned record emits its own `<Entity>.Updated` + audit **under the offboard correlationId**.
   *Touches:* `src/features/{leads,customers,tickets}`, `src/shared/data/LocalStorageRepository.ts`, `src/shared/events/{bus.ts,auditLog.ts,correlation.ts}`, `src/shared/ui/components/ConfirmDialog` ¬∑ *Seam:* batched `Repository<T>.update` (4-beat per record) + correlationId ¬∑ *Schema/Map:* `Lead.Updated`/`Customer.Updated`/`Ticket.Updated` (canonical).
4. **AC4 ‚Äî Emits Tenant.SubsidiaryRemoved + audit, sharing the correlationId (UC-2).** The subsidiary soft-delete emits one `Tenant.SubsidiaryRemoved` `DomainEvent` + one `AuditEvent`, both threaded with the **same** offboard correlationId as all reassignment events.
   *Touches:* `src/shared/events/{bus.ts,auditLog.ts,eventTypes.ts,correlation.ts}` ¬∑ *Seam:* 4-beat emit + audit ¬∑ *Schema/Map:* `Tenant.SubsidiaryRemoved` (canonical).
5. **AC5 ‚Äî ConfirmDialog required (destructive).** The destructive offboard requires explicit confirmation via the shared `ConfirmDialog`, which also surfaces the reassignment-target selector before commit.
   *Touches:* `src/shared/ui/components/ConfirmDialog` ¬∑ *Seam:* destructive-action confirmation ¬∑ *Schema/Map:* n/a.
6. **AC6 ‚Äî Soft-deleted excluded from lists unless includeDeleted=true.** Per constitution ¬ß4, lists exclude soft-deleted rows unless `filter.includeDeleted=true`; the offboarded subsidiary disappears from the switcher and lists by default.
   *Touches:* `src/shared/data/LocalStorageRepository.ts` ¬∑ *Seam:* `Repository<T>.list` filter ¬∑ *Schema/Map:* `Page<T>`/list filter.

## UX & Behavior (from prototype)
*Source: `prototype/app/tenancy.jsx` (`OffboardDialog`), `prototype/app/store.jsx` (`commitOffboard`). The marquee tenancy edge flow ‚Äî build identical to the prototype.*

- **`OffboardDialog` = danger `ConfirmDialog` with a batch-reassign saga.** Phases `choose ‚Üí running`.
- **Impact preview:** counts of **active** records that must move ‚Äî **Leads / Customers / Tickets** (only non-terminal records count; `ACTIVE_STATES` excludes converted/churned/closed).
- **Required field "Reassign active records to":** another **active** subsidiary in the same tenant **or "Parent level (shared)"**. Plus a **"Simulate a mid-batch failure"** toggle.
- **Run (success):** progress bar ticks **`done / total`** at ~`max(90, base/2)` per record. On commit ‚Üí soft-deletes the sub, reassigns each active record's `subsidiaryId` to the target and re-scopes its `owner`/`assignee` to the target's default person; **success toast** *"<sub> offboarded. N active records reassigned to <target>."* If the current scope was the offboarded sub, scope resets to tenant.
- **Run (mid-batch fault):** at **~60%** the run aborts, **nothing is committed**, a **persistent danger toast** fires ‚Äî *"Reassignment failed mid-batch ‚Äî rolled back. No records moved. <sub> is still active."* ‚Äî with **Try again**, and the dialog closes.
- **ConfirmDialog discipline:** danger-toned; initial focus on the **safe Cancel** control (never the destructive button); **`Esc` is suppressed while running**.
- **Motion:** dialog enter `--crm-base`; progress ticks ~`max(90, base/2)`/record; failed-run rollback is the loud reversal (toast `--crm-fast`, persistent).

## Inherited Universal Conformance (subset)
- **UC-1 ‚Äî Four UI states.** Offboard dialog + post-offboard lists render all four states; the batch operation shows progress/error and rolls back optimistically on failure.
- **UC-2 ‚Äî Dual events, one correlationId.** Every reassignment `<Entity>.Updated` and the final `Tenant.SubsidiaryRemoved` share one offboard correlationId, each paired with its audit event.
- **UC-5 ‚Äî Tenant scope.** Reassignment targets are constrained to the **same tenant** (another active subsidiary or parent-level); cross-tenant reassignment is impossible (scope from context).
- **TC ‚Äî Traceability.** story ‚Üí spec ‚Üí code ‚Üí test ‚Üí `Closes #`; preview green; `sprint-status.yaml`; passes `bmad-code-review`.

## Test Requirements (NFR-12)
- **Vitest:** soft-delete sets `deletedAt` (never hard-deletes); reassignment moves only **active** records, sets `subsidiaryId` to target and re-scopes `ownerId`/`assigneeId`; one correlationId spans all reassignment `<Entity>.Updated` + the `Tenant.SubsidiaryRemoved`; target validation rejects an inactive/other-tenant subsidiary; permission cell ‚Äî only tenant_admin may offboard; hard-delete never granted.
- **RTL:** ConfirmDialog blocks until confirmed and requires a target selection; four states on the batch; optimistic rollback when a reassignment fails mid-batch.
- **Playwright:** offboard happy path (admin picks target ‚Üí records reassigned, subsidiary leaves switcher/lists, reappears with `includeDeleted=true`); **cross-tenant isolation** ‚Äî reassignment target picker offers only same-tenant active subsidiaries + parent-level, never another tenant's subsidiaries; offboarded data never leaks cross-tenant.
